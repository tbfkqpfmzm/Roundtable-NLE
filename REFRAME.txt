1.  DESIGN PHILOSOPHY & KEY DEVIATIONS FROM SOURCE PLAN
================================================================================

The AI's plan was directionally sound but made several assumptions that do
not match the ROUNDTABLE codebase.  The following deviations are critical:

  DEVIATION 1 — No OpenCV dependency
    The source plan assumes OpenCV for frame manipulation.  ROUNDTABLE does
    NOT ship OpenCV, and adding a 300 MB+ dependency is against project
    philosophy (all deps are minimal / FetchContent).  Instead:
      • Frame sampling uses the existing VideoDecoder directly.
      • Face detection input: raw RGB frames from VideoDecoder.
      • Image resize (for detector input): stb_image_resize.h (already
        available via third_party/stb).  No OpenCV needed.

  DEVIATION 2 — No MediaPipe framework dependency
    The source plan suggests the full MediaPipe framework.  This is
    a massive dependency.  Instead, use a standalone BlazeFace TFLite
    model loaded via a minimal C++ TFLite-era runtime or pure-C++ header-
    only face detection.  Specifically:
      • Ship the pre-trained BlazeFace TFLite model in assets/.
      • Use a lightweight TFLite interpreter (tensorflow/lite/c/c_api.h)
        or a minimal C++ BlazeFace implementation.
      • Alternative: use the existing omnishotcut Python subprocess pattern
        for detection, keeping C++ clean.  (See Step 3 for trade-offs.)

  DEVIATION 3 — Keyframes go DIRECTLY onto Clip tracks, not CSV files
    The source plan suggests CSV keyframe files.  ROUNDTABLE already has a
    complete keyframe system:
      • Clip::positionX() / positionY() — KeyframeTrack<float>
      • Clip::scaleX() / scaleY() — KeyframeTrack<float>
      • Clip::rotation() — KeyframeTrack<float>
    Auto-reframe generates keyframes that directly populate these tracks.
    This means:
      • No separate file format
      • Keyframes are immediately editable in the Curve Editor
      • Auto-reframe can be disabled by deleting the generated keyframes
      • Undo/redo works natively through CommandStack

  DEVIATION 4 — AutoReframe as a non-GPU Effect, not a separate module
    The source plan suggests an entirely separate module pipeline.  Instead,
    auto-reframe fits naturally as:
      • An Effect (EffectType::AutoReframe) that stores analysis state
        (detection results, keyframe data) and can be toggled on/off.
      • The effect does NOT have a GPU compute shader — it is CPU-only
        analysis.  It generates keyframes on the Clip's transform tracks.
      • When disabled, generated keyframes remain but are ignored.
      • When re-enabled, it can either reuse cached analysis or re-run.

  DEVIATION 5 — Use existing async pattern (OmniShotDetector)
    ROUNDTABLE already has std::thread + atomic cancel + callback pattern
    in OmniShotDetector.  Reuse this pattern for the analysis worker.

  DEVIATION 6 — Compositor integration
    The auto-reframe crop is applied by writing to the Clip's transform
    tracks, which are already consumed by the Compositor via
    CompositeServiceLayerBuild.  No compositor changes needed, UNLESS
    we want a "render-time" reframe that doesn't modify source clips.
    For V1, modifying clip transform tracks is the right approach
    (user-editable by design).

  namespace rt::autoreframe {

  struct Vec2 { float x, y; };

  struct Rect {
      float x, y;       // Top-left
      float w, h;       // Width, height
  };

  struct FaceDetection {
      Rect       bbox;
      float      confidence{0.0f};
      Vec2      keypoints[6]{};   // BlazeFace: eyes, ears, nose, mouth
  };

  struct TrackedFace {
      int        id{-1};
      Rect       bbox;
      Vec2      center;
      int        age{0};          // Frames since last detection
      int        lifetime{0};     // Total frames tracked
  };

  struct CameraState {
      Vec2  center{0.5f, 0.5f};  // Normalized crop center (0-1 UV space)
      float zoom{1.0f};           // 1.0 = full frame, >1 = zoom in
  };

  struct ReframeKeyframe {
      double time{0.0};           // Seconds (clip-local time)
      Vec2   center;
      float  zoom{1.0f};
  };

  /// Analysis result: what the pipeline produces
  struct ReframeAnalysisResult {
      std::vector<ReframeKeyframe> keyframes;
      std::vector<ReframeKeyframe> rawCameraPath;  // Unsmooth path for debug
      std::vector<DetectedCut>     shotCuts;        // Reuse from OmniShotDetector
  };

  /// Framing strength preset
  enum class FramingStrength : uint8_t {
      Conservative,   // Slow, stable, minimal movement
      Balanced,       // Default
      Aggressive      // Tighter crop, faster response
  };

  } // namespace rt::autoreframe

NOTES:
  - Uses float UV coordinates (0-1) throughout, matching the existing
    compositor's UV-space transform convention.
  - ReframeKeyframe is SEPARATE from the existing Keyframe<T> template.
    It represents the analysis output BEFORE conversion to clip tracks.
    Step 11 handles conversion.

  namespace rt::autoreframe {

  class FrameSampler {
  public:
      FrameSampler();
      ~FrameSampler();

      /// Open a video file for sampling. Returns false on failure.
      bool open(const std::filesystem::path& path);

      /// Close and release decoder.
      void close();

      /// Get stream info (width, height, fps, duration, frame count).
      const VideoStreamInfo& info() const;

      /// Decode and return an RGB frame at the given time (seconds).
      /// Returns empty vector on failure.
      /// Uses the existing VideoDecoder::decodeAt() internally.
      std::vector<uint8_t> decodeFrame(double timeSeconds);

      /// Decode a frame, resize to target dimensions (for detector input).
      /// Uses stb_image_resize.h for the resize.
      std::vector<uint8_t> decodeFrameResized(double timeSeconds,
                                                int targetW, int targetH);

      /// Get raw width/height of decoded frames.
      int width() const;
      int height() const;

      /// Number of frames in the source.
      int64_t frameCount() const;

  private:
      std::unique_ptr<VideoDecoder> m_decoder;
      std::vector<uint8_t>          m_rgbBuffer;   // Reusable RGB buffer
      std::vector<uint8_t>          m_resizeBuffer; // Reusable resize buffer
  };

  } // namespace rt::autoreframe

KEY DESIGN DECISIONS:
  - Allocates a new VideoDecoder per analysis run.  The decoder is
    independent from the playback decoder — it's used only during
    analysis.
  - Converts to RGB on the CPU using FFmpeg's sws_scale (already used
    in VideoDecoder for format conversion).
  - stb_image_resize.h (header-only, already in third_party) resizes
    frames to the detector's input size (e.g. 128×128 or 256×256).
  - Frame skipping: only samples every N frames (N=5 default). The
    caller decides the stride; FrameSampler just seeks and decodes.

  class BlazeFaceDetector : public IFaceDetector {
      // Standalone C++ BlazeFace implementation.
      // Loads the pre-trained .tflite model on init() using flatbuffers
      // (already available via vcpkg), runs inference via a minimal TFLite
      // model interpreter, and decodes the anchor-box output.
      //
      // MODEL SOURCE: Use PINTO's exported BlazeFace TFLite models:
      //   https://github.com/PINTO0309/PINTO_model_zoo/tree/main/079_BlazeFace_tf2
      // These are pre-exported, optimized TFLite models ready for inference.
      // Download blazeface_back.tflite and blazeface_front.tflite, ship in
      // assets/models/.
      //
      // C++ CODE: The standalone inference code (~500 lines) must be written
      // from scratch or adapted from an open-source BlazeFace C++ port.
      # Reference implementations:
      #   - https://github.com/waittim/BlazeFace-CPP (standalone, MIT)
      #   - https://github.com/tensorflow/tflite-micro (has BlazeFace runner)
      # The C++ code needs only: model loader, input tensor setup, inference
      # invocation, anchor decoding, and non-max suppression.
      #
      // WHY NOT tflite-micro: the full TFLM repo includes Xtensa kernels,
      // ARM CMSIS-NN optimizations, and FlatBuffers code generation that
      // conflicts with vcpkg's FlatBuffers.  Build-time cost far outweighs
      // the runtime benefit for a ~5ms/inference face detector.
      //
      // Unit tests: load test PNG images via stb_image.h (already in
      // third_party/stb) to verify detection on known face/non-face patterns.
  };

DETECTION FREQUENCY:
  - Run detection every 5th frame (configurable).
  - Between detections, the tracker interpolates (Step 4).
  - At 30 fps source: 6 detections/second.  BlazeFace runs <5ms per
    inference on CPU, so total analysis <30ms/second of source.

7.  STEP 4 — SIMPLE FACE TRACKING
================================================================================

File: src/core/autoreframe/tracking/FaceTracker.h/.cpp

  namespace rt::autoreframe {

  class FaceTracker {
  public:
      FaceTracker();

      /// Process a set of face detections from the current frame.
      /// Assigns stable IDs by matching nearest detections to existing tracks.
      /// Called every N frames (when detection runs).
      void update(const std::vector<FaceDetection>& detections,
                  int frameIndex);

      /// Interpolate tracked positions for frames between detections.
      /// Called on frames where detection did NOT run.
      void interpolate(int frameIndex);

      /// Get the list of currently tracked faces.
      const std::vector<TrackedFace>& trackedFaces() const;

      /// Get the primary subject (largest / most central / longest-lived).
      const TrackedFace* primarySubject() const;

      /// Reset all tracks (e.g. on shot cut).
      void reset();

  private:
      struct Track {
          int            id;
          TrackedFace    lastDetection;
          int            detectionInterval; // Detection frequency in frames
      };

      std::vector<Track> m_tracks;
      int                m_nextId{1};
      int                m_lastDetectionFrame{-1};

      /// Match detections to existing tracks using nearest-centroid.
      /// Returns assignment pairs (trackIndex, detectionIndex).
      std::vector<std::pair<int,int>> matchTracks(
          const std::vector<FaceDetection>& detections);

      /// Remove tracks that haven't been matched for N frames.
      void pruneStaleTracks(int maxAge = 30);
  };

  } // namespace rt::autoreframe

MATCHING STRATEGY (NOT ByteTrack):
  - Hungarian algorithm on centroid distance matrix (size <= 10, trivial).
  - Distance threshold: 20% of frame width (configurable).
  - Tracks older than 30 frames (~1 sec) without a detection are pruned.
  - Interpolation: linear between last two detected positions.

This is deliberately simple.  For talking-head content with single faces,
there is almost never ambiguity to resolve.


================================================================================
8.  STEP 5 — SUBJECT SELECTION HEURISTICS
================================================================================

Implemented inside CameraSolver (Step 6), not as a separate class.

  Subject priority (for multi-face scenarios):
    1. Largest face (by bbox area) → primary subject
    2. In case of tie: face closest to frame center
    3. In case of tie: longest-visible face (highest lifetime)

  Multi-face fallback:
    If 2+ faces are detected and all are roughly similar in size
    (major face <= 2× minor face area), compute the UNION bounding
    box of all faces.  The virtual camera frames all faces together.

  Face count change handling:
    - If a new face appears (wasn't in previous frame), don't jump.
      Wait 5 frames to confirm stability, then smoothly transition.
    - If a face disappears, don't snap the crop.  Blend the camera
      center toward the remaining face over ~0.5 seconds.

  namespace rt::autoreframe {

  struct SolverConfig {
      // Composition
      float headroom{0.1f};         // Space above head (fraction of frame height)
      float eyeLine{0.33f};         // Eye position in upper third (UV vertical)
      float deadZone{0.10f};        // 10% dead zone (fraction of frame)
      float maxMoveSpeed{0.15f};    // Max center movement per second (UV)
      float maxZoomSpeed{2.0f};     // Max zoom change per second
      float minZoom{1.0f};          // Minimum zoom (1.0 = full frame)
      float maxZoom{2.5f};          // Maximum zoom (2.5× crop)
      float subjectWeight{0.7f};    // How much subject position affects crop
      float faceWeight{0.3f};       // How much face position within subject
  };

  class CameraSolver {
  public:
      CameraSolver();
      explicit CameraSolver(const SolverConfig& config);

      /// Set target aspect ratio.  E.g. 9.0f/16.0f for vertical video.
      void setTargetAspect(float aspect);

      /// Set framing strength preset.
      void setFramingStrength(FramingStrength strength);

      /// Solve camera state for a frame given tracked face data.
      /// Call AFTER spring smoothing (the solver computes the "target" state).
      CameraState solveTarget(const TrackedFace& subject,
                              const std::vector<TrackedFace>& allFaces,
                              double timeSeconds,
                              int sourceW, int sourceH);

      /// Access the raw solver config (for UI preview).
      SolverConfig& config() { return m_config; }
      const SolverConfig& config() const { return m_config; }

  private:
      SolverConfig m_config;
      float        m_targetAspect{9.0f / 16.0f};
      FramingStrength m_strength{FramingStrength::Balanced};
  };

  } // namespace rt::autoreframe

COMPOSITION RULES:
  1. HEADROOM: Place the subject's head in the upper third of the frame.
     Specifically, the top of the face bbox sits at ~10% from the top
     of the cropped output frame.

  2. EYE LINE: The face center should sit near the upper-third horizontal
     line (y ≈ 0.33 in normalized output UV).

  3. LEAD ROOM: If the subject is looking left/right, add offset in the
     look direction proportional to the face yaw (estimated from eye
     asymmetry).  For V1, this is optional — skip if face keypoints
     are unavailable.

  4. MAX MOVE SPEED: The camera center must NOT move faster than
     `maxMoveSpeed` UV units per second.  This prevents jarring jumps
     when detection flickers.

  5. The solver computes the DESIRED (target) camera state.  Spring
     smoothing (Step 8) applies the gradual motion.

10. STEP 7 — DEAD ZONE LOGIC
================================================================================

Integrated into CameraSolver::solveTarget().

  ALGORITHM:
    1. Compute the raw desired center (face center projected into output UV).
    2. Compute delta = rawCenter - currentCameraCenter.
    3. If |delta.x| < deadZone AND |delta.y| < deadZone:
         → return current camera center unchanged (no movement).
    4. Otherwise:
         → Apply movement scaled by (1 - deadZone) factor.
           e.g. effectiveDelta = delta * (|delta| - deadZone) / |delta|

  This is CRITICAL for preventing sub-pixel jitter when the subject is
  sitting still.  The dead zone acts as a hysteresis band.

  DEAD ZONE SIZE by preset:
    - Conservative: 12%
    - Balanced:     10%
    - Aggressive:   6%

  ⚠ CRITICAL ORDERING: The dead zone must compare against the SPRING'S
  CURRENT OUTPUT, not the raw target.  Correct data flow:

     RawFacePos
         │
         ▼
     CameraSolver::solveTarget()  →  computes RawTarget (cx, cy, zoom)
         │
         ▼
     *** DEAD ZONE CHECK ***
     delta = RawTarget - Spring.current()
     if |delta| < deadZone:
         AdjustedTarget = Spring.current()   ← NO MOVEMENT
     else:
         AdjustedTarget = RawTarget          ← ALLOW MOVEMENT
         │
         ▼
     SpringDamper::update(AdjustedTarget, dt)
         │
         ▼
     SmoothedCameraState  →  stored as keyframe

     If applied in the wrong order (dead zone before spring), the spring
     will smoothly drift INTO the dead zone boundary, and the camera will
     never settle at the correct position — it will hover at the edge of
     the dead zone instead.

11. STEP 8 — SPRING SMOOTHING
================================================================================

File: src/core/autoreframe/camera/SpringDamper.h (header-only)

  namespace rt::autoreframe {

  /// Critically damped spring smoother.
  /// Smooths a 1D or 2D value toward a target with natural-feeling motion.
  /// No oscillation in critically damped mode.
  template <typename T>
  class SpringDamper {
  public:
      SpringDamper(float stiffness = 8.0f, float damping = 0.85f)
          : m_stiffness(stiffness), m_damping(damping) {}

      /// Set spring parameters.
      void setStiffness(float s) { m_stiffness = s; }
      void setDamping(float d)   { m_damping = d; }

      /// Update the spring toward target. Returns the smoothed value.
      /// dt = delta time in seconds.
      T update(const T& target, float dt);

      /// Snap directly to a value (reset velocity to zero).
      void snapTo(const T& value);

      /// Get current value without updating.
      const T& current() const { return m_current; }

      /// Get current velocity.
      const T& velocity() const { return m_velocity; }

      /// Reset all state.
      void reset();

  private:
      float m_stiffness{8.0f};
      float m_damping{0.85f};
      T     m_current{};
      T     m_velocity{};
  };

  // Vec2 specialization
  template <>
  inline Vec2 SpringDamper<Vec2>::update(const Vec2& target, float dt) {
      // Spring force
      Vec2 spring = {
          (target.x - m_current.x) * m_stiffness,
          (target.y - m_current.y) * m_stiffness
      };
      // Apply damping to velocity
      float dampingFactor = std::pow(m_damping, dt * 60.0f);  // Frame-rate independent
      m_velocity.x = (m_velocity.x + spring.x * dt) * dampingFactor;
      m_velocity.y = (m_velocity.y + spring.y * dt) * dampingFactor;
      // Integrate position
      m_current.x += m_velocity.x * dt;
      m_current.y += m_velocity.y * dt;
      return m_current;
  }

  } // namespace rt::autoreframe

SPRING PARAMETERS by preset:
  ┌──────────────┬───────────┬──────────┐
  │ Preset       │ Stiffness │ Damping  │
  ├──────────────┼───────────┼──────────┤
  │ Conservative │    4.0    │   0.90   │
  │ Balanced     │    8.0    │   0.85   │
  │ Aggressive   │   15.0    │   0.80   │
  └──────────────┴───────────┴──────────┘

  Separate springs for: center.x, center.y, zoom (float).


================================================================================
12. STEP 9 — DYNAMIC ZOOM LOGIC
================================================================================

Implemented inside CameraSolver.

  ZOOM RULES:
    1. SINGLE FACE: Zoom to frame the face with headroom margin.
       targetZoom = min(maxZoom, max(w_target / w_face, h_target / h_face))
       where (w_target, h_target) is the output resolution and the face
       is padded by headroom + chin room (≈20% total margin).

    2. MULTIPLE FACES: Zoom out so the union bbox of all faces fits with
       15% margin on each side.

    3. MOTION ZOOM: If subject velocity (in source pixels/sec) exceeds
       a threshold, zoom out slightly (multiply zoom by 0.95).  This
       reduces perceived jitter during rapid movement.  Recover zoom
       when velocity drops below threshold.

    4. CONSTRAINTS:
       - Clamp zoom: [minZoom, maxZoom]
       - Clamp zoom velocity: [−maxZoomSpeed, +maxZoomSpeed] per second
       - Never change zoom by more than 0.3× in a single frame

    5. AVOID CONSTANT ZOOM CHANGES:
       - If current zoom differs from target by < 0.05, don't move.
       - This prevents breathing artifacts ("micro-zooms").

KEYFRAME CONVERSION:
  The CameraState (normalized center + zoom) must be converted to
  Clip transform parameters.  IMPORTANT DECISION: Do we write to the
  clip's built-in transform (positionX/Y, scaleX/Y) or add a
  Transform2D effect with keyframes?

  RECOMMENDATION: Use a dedicated Transform2D effect created by the
  auto-reframe system, with AnchorX=0, AnchorY=0.  This effect is SEPARATE
  from any Transform2D the user may have added manually.

  ⚠ OWNERSHIP CONFLICT: If the user already has a Transform2D effect in
  their stack (for manual positioning/crop), auto-reframe MUST NOT
  overwrite its keyframes.  Instead:
    1. Add a bool marker to Effect: m_isAutoReframeTarget{false}.
    2. Auto-reframe creates a NEW Transform2D effect with this flag set.
    3. Two Transform2Ds in the stack compose naturally: user's on top
       (manual tweaks), auto-reframe's below (reframe crop).
    4. The dedicated effect is identified by the flag and hidden from
       the user's effect list (or shown with a lock icon).
    5. When auto-reframe is disabled, this dedicated effect is toggled
       off (not removed), preserving user adjustments above it.

  Rationale for the dedicated effect:
    - The clip's built-in positionX/Y is also used for manual positioning.
      Auto-reframe would overwrite user adjustments.
    - A Transform2D effect sits in the effect stack and can be enabled/
      disabled independently.  The clip's built-in transform stays clean.
    - The anchor point at (0,0) means scale happens from top-left,
      which simplifies the crop math (no anchor-offset compensation).
    - The Transform2D effect owns OffsetX/Y, Scale params — these are
      exactly what auto-reframe needs.

  Conversion formulas (from camera state → Transform2D params):

    Given:
      - sourceW × sourceH = original frame size (e.g. 1920×1080)
      - targetAspect = targetW / targetH (e.g. 9/16 = 0.5625)
      - camera.center = (cx, cy) in UV [0,1]
      - camera.zoom = z (1.0 = full source)

    The crop rectangle in source pixel space:
      cropH = sourceH / z
      cropW = cropH * targetAspect
      cropX = (cx * sourceW) - cropW / 2
      cropY = (cy * sourceH) - cropH / 2

    Transform2D parameters (AnchorX=0, AnchorY=0):
      Scale  = sourceW / cropW   (uniform scale, maintain aspect via cropW/H ratio)
      OffsetX = -cropX
      OffsetY = -cropY

    ⚠ BOUNDS CHECK: Transform2D clamps Scale to [0.01, 10.0] and OffsetX/Y
    to [-2000, 2000].  If auto-reframe produces values outside these ranges
    (e.g. a very small face producing Scale > 10), the crop will be wrong
    and the error invisible to the user.  applyKeyframes() MUST clamp
    generated values to these bounds BEFORE writing keyframes.

    ⚠ HIGH RISK: These formulas MUST be validated against the actual
    compositor shader coordinate system (CompositeServiceLayerBuild.cpp).
    The existing compositor uses UV-space transform matrices.  If the
    math is wrong, the crop will show black bars or wrong framing.
    Write a unit test that:
      1. Creates a known test pattern (e.g. 1920×1080 with a crosshair)
      2. Applies the transform
      3. Reads back the output
      4. Verifies the crosshair is at the expected position
    Do this BEFORE integrating with the UI.

    The generated keyframes are written via CommandStack:
      auto cmd = std::make_unique<CompoundCommand>("Auto Reframe");
      for each keyframe:
          cmd->addChild(std::make_unique<SetKeyframeCommand>(
              clip, Transform2D::Param::OffsetX, tick, value));
          ...
      commandStack.execute(std::move(cmd));

    This ensures:
      - Thread safety: CommandStack serializes all model mutations
      - Undo/redo: user can undo the entire reframe operation
      - No data races with the compositor render thread

KEYFRAME STRIDE:
  - Generate one keyframe every 2-4 video frames, NOT every frame.
    Bezier interpolation makes this imperceptible — 7,500 keyframes for
    a 10-minute clip at 30fps is visually indistinguishable from 18,000.
  - More importantly, inserting 18,000 keyframes via addKeyframe() is
    O(n²): each call does lower_bound (O(log n)) + vector::insert (O(n)
    shift).  162 million element shifts would freeze the UI.
  - Instead, add a batchInsert() method to KeyframeTrack:
      void batchInsert(std::span<const Keyframe<T>> keyframes);
    This appends all keyframes, then does a single std::sort + optional
    dedup.  O(n log n) total — ~100k comparisons vs 162M shifts.

INTERPOLATION MODE:
  - Use InterpMode::Bezier with default handles (0.33, 0.0, 0.67, 1.0)
    for smooth ease-in/out between keyframes.
  - DO NOT use Linear — it produces visible corner artifacts in camera
    motion.

  namespace rt::autoreframe {

  // Callbacks (matching the pattern from OmniShotDetector)
  using ReframeProgressFn = std::function<void(float percent, int64_t frame, int64_t total)>;
  using ReframeCompleteFn = std::function<void(ReframeAnalysisResult result)>;
  using ReframeErrorFn    = std::function<void(std::string message)>;

  /// Top-level orchestrator for auto-reframe analysis.
  /// Follows the same async pattern as OmniShotDetector.
  class AutoReframePipeline {
  public:
      AutoReframePipeline();
      ~AutoReframePipeline();

      // Non-copyable
      AutoReframePipeline(const AutoReframePipeline&) = delete;
      AutoReframePipeline& operator=(const AutoReframePipeline&) = delete;

      /// Start async analysis of a video clip.
      /// \param sourcePath   Path to the video file
      /// \param startTime    Source in point (seconds)
      /// \param endTime      Source out point (seconds)
      /// \param targetAspect Desired output aspect ratio (e.g. 9.0/16.0)
      /// \param strength     Framing strength preset
      /// \param onProgress   Progress callback (main thread, use QMetaObject::invoke)
      /// \param onComplete   Completion callback
      /// \param onError      Error callback
      void analyzeAsync(const std::filesystem::path& sourcePath,
                        double startTime, double endTime,
                        float targetAspect,
                        FramingStrength strength,
                        ReframeProgressFn onProgress,
                        ReframeCompleteFn onComplete,
                        ReframeErrorFn onError);

      /// Cancel a running analysis.
      /// On cancel, onComplete is called with partial results
      /// (whatever keyframes were generated up to the cancel point).
      /// The user keeps partial keyframes rather than losing all work.
      void cancel();

      /// Is analysis currently running?
      bool isRunning() const { return m_running.load(); }

  private:
      void workerFunc(std::filesystem::path sourcePath,
                      double startTime, double endTime,
                      float targetAspect, FramingStrength strength,
                      ReframeProgressFn onProgress,
                      ReframeCompleteFn onComplete,
                      ReframeErrorFn onError);

      std::thread       m_worker;
      std::atomic<bool> m_cancelled{false};
      std::atomic<bool> m_running{false};
  };

  } // namespace rt::autoreframe

WORKER THREAD PIPELINE:
  ┌─────────────────┐
  │ FrameSampler     │  ← Opens decoder, seeks to startTime
  │ (worker thread)  │
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │ ShotDetector     │  ← Histogram diff between every Nth frame
  │ (worker thread)  │
  └────────┬────────┘
           │
           ▼  ┌──────────────────────────────┐
              │  Detection Loop:              │
              │  for each Nth frame:          │
              │    1. Decode + resize frame   │
              │    2. BlazeFace detect        │
              │    3. FaceTracker::update()    │
              │  for intermediate frames:     │
              │    4. FaceTracker::interpolate()│
              │    5. CameraSolver::solve()   │
              │    6. SpringDamper::update()  │
              └──────────────────────────────┘
           │
           ▼
  ┌─────────────────┐
  │ Keyframe Gen     │  ← Convert camera path to clip keyframes
  │ (worker → main)  │
  └─────────────────┘

  - The worker thread does ALL analysis (decode + detect + solve).
  - applyKeyframes() must run on the main thread (it modifies the
    clip data model, which the UI reads).
  - Progress: reported via callback every ~5%.  The onProgress callback
    also delivers intermediate ReframeKeyframe data so the UI can show
    partial framing in the curve editor during analysis.
  - CANCELATION: When cancel() is called, the worker thread finishes its
    current frame, then calls onComplete with whatever keyframes were
    generated so far (partial results).  The UI keeps these partial
    keyframes rather than discarding them.

19. STEP 16 — PERFORMANCE TARGETS
================================================================================

  ┌────────────────────────────────────────────────────────────────────┐
  │ TARGET (1080p source → 9:16 output):                              │
  │   • Analysis: ≥ 2× realtime on laptop CPU (i7-12700H)            │
  │     → 30 fps source analyzed in ≤15 seconds per minute           │
  │   • Detection: BlazeFace ≤5ms per 128×128 frame on CPU           │
  │   • Memory: < 200 MB peak (no frame storage, decode → detect →   │
  │     discard)                                                      │
  │   • UI: never blocked during analysis (worker thread)            │
  │   • Keyframe generation: < 100ms for 1 hour of footage           │
  └────────────────────────────────────────────────────────────────────┘

  OPTIMIZATION STRATEGIES:
    1. FRAME SKIPPING: Detect every 5th frame.  At 30 fps → 6 fps
       detection rate.  BlazeFace at 128×128: ~3ms → 18ms/sec of
       source → 56× realtime detection speed.

    2. SEQUENTIAL DECODE (CRITICAL): DO NOT seek for each sampled
       frame.  Seeking to every 5th frame via FFmpeg is slow (~30-100ms
       per seek for H.264 with 2s GOPs — 3,600 seeks × 50ms = 3 minutes
       for a 10-min clip).  Instead, decode sequentially and call
       decodeNext() 5× per sampled frame.  This keeps the decoder hot
       in the hardware path (NVDEC: <1ms per frame) and avoids seeks.
       The FrameSampler's API must support sequential iteration, not
       just random-access decodeFrame(timeSeconds).

    3. ZERO-COPY: The worker's VideoDecoder stays in software mode
       (forceSoftware=true) to keep CPU-side frame data accessible.
       The playback decoder continues using NVDEC unaffected.

    4. REUSE FRAME BUFFERS: FrameSampler reuses its RGB and resize
       buffers across frames (no per-frame allocation).

    5. EARLY TERMINATION: If the clip is very long (e.g. 2 hours of
       podcast), consider adaptive sampling: detect every 10th frame
       during static segments, every 3rd during movement.

    6. PROGRESSIVE RESULTS: Show partial keyframes in the timeline
       as analysis progresses (update keyframes every 1 second of
       analyzed content).  This gives the user immediate feedback.

APPENDIX C — KNOWN GAPS & RISK MITIGATIONS
================================================================================

  GAP 1 (HIGH) — BlazeFace inference code does not exist yet
  ──────────────────────────────────────────────────────────────────────────────
    The plan assumes "~500 lines of C++" to load a TFLite model, run inference,
    decode BlazeFace anchor boxes, and apply non-max suppression.  This code
    must be written from scratch or ported from an existing implementation.

    What's involved:
      • Load .tflite flatbuffer model and validate input/output tensor shapes
      • Set up input tensor: 128×128×3 RGB, normalized to [-1, 1]
      • Run inference (flatbuffers tensor access — no TF runtime needed)
      • Decode 17 anchor boxes from regression outputs:
        Each anchor outputs 16 regression values (4 box corners × 4 coords)
        and 1 confidence score.  Apply sigmoid to score, decode box centers
        using anchor prior positions (29 predefined priors per BlazeFace spec).
      • Non-max suppression with IoU threshold (default 0.5)
      • Return FaceDetection structs

    Risk: If ANY step is wrong, detection silently fails.  Debugging without
    visual output is extremely difficult.

    Mitigation:
      • FIRST deliverable of Phase 2: a STANDALONE CONSOLE BINARY
        (not integrated into the NLE) that:
          - Loads a test PNG via stb_image
          - Runs BlazeFace
          - Saves an output PNG with bounding boxes drawn on it
        This binary proves the inference pipeline works before any other
        code is written.  Do not proceed until this binary produces correct
        detections on known test images (face_front.png, face_side.png).
      • Expected time: 2-3 days for the standalone binary alone.
        Budget 4-6 days total for Phase 2 (not 3-4 as originally estimated).


  GAP 2 (MEDIUM) — Clip trimming silently invalidates results
  ──────────────────────────────────────────────────────────────────────────────
    Auto-reframe analysis is computed relative to the clip's sourceIn and
    duration.  If the user trims the clip after analysis, the generated
    keyframes cover the wrong time range.

    What's affected:
      • Keyframe timing shifts relative to the new visible range.
      • The crop framing at the edges may be incorrect (face may be
        partially out of frame at the new in/out points).
      • User has no visual indication that results are stale.

    Mitigation (addressed in Step 15D):
      • AutoReframeEffect registers as an observer on the Clip.
      • On sourceIn or duration change: set analysisDirty = true.
      • UI shows a "needs re-analysis" badge (yellow triangle or similar).
      • Old keyframes are NOT discarded — the user can still see the
        previous framing and decide when to re-run.
      • Implementation pattern: Clip emits a signal or calls a callback
        on property change.  Hook into the existing ChangeManager or
        TimelineObserver pattern (see ExportPanel for reference).


  GAP 3 (LOW) — "Remove Auto Reframe" cleanup is underspecified
  ──────────────────────────────────────────────────────────────────────────────
    When the user removes auto-reframe from a clip (via the panel's × button
    or the context menu), multiple objects must be deleted atomically:

      • AutoReframeEffect itself (from EffectStack)
      • The dedicated Transform2D effect (m_isAutoReframeTarget=true)
      • All keyframes on that Transform2D's OffsetX, OffsetY, Scale tracks
      • The analysis cache entry (optional — can be lazy-cleaned)

    If only the AutoReframeEffect is removed, the orphaned Transform2D
    effect and its keyframes remain active, warping the clip's rendering
    with no indication of why.

    Mitigation (addressed in Phase 4 task 8):
      • Wrap all three deletions in a single CompoundCommand:
          auto cmd = CompoundCommand("Remove Auto Reframe");
          cmd->addChild(RemoveEffectCommand(clip, autoReframeEffectIndex));
          cmd->addChild(RemoveEffectCommand(clip, dedicatedTransform2DIndex));
          cmd->addChild(ClearKeyframesCommand(clip.transform2D->params()));
          commandStack.execute(cmd);
      • This gives atomic undo: one Ctrl+Z restores everything.
      • The dedicated Transform2D is identified by m_isAutoReframeTarget
        flag.  Only the flagged Transform2D is removed — user-added
        Transform2D effects are untouched.
