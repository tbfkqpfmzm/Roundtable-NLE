     CPU safe mode).

  ❌ Parallel pipeline stages — All work is serial (upload→barrier→

     overlap between decode, upload, and composite of different frames.

  ❌ Unified resource management — Three separate caches:
     1. CPU FrameCache (RAM ring buffer per media source)
     2. GpuTextureCache (VRAM LRU inside CompositeEngine)
     3. Composite result LRU (inside CompositeEngine)
     No coordination between them; no shared eviction policy.

  ❌ GPU timestamp queries — No telemetry on per-pass GPU time.
     Only CPU-side high_resolution_clock timing around the whole
     composite() call.

  ❌ CpuReadback as explicit pass — Currently baked into compositor's
     recordReadback() call. Not a separate, isolatable pass.

B. TARGET ARCHITECTURE OVERVIEW
================================================================================

B.1 The Render Graph DAG Concept
─────────────────────────────────

  A render graph is a Directed Acyclic Graph where each node is a render
  pass with explicit declared inputs, outputs, and resource transitions.
  The graph dispatcher walks the DAG in topological order, executing each
  pass.  Automatic pipeline barriers between passes ensure correct
  synchronization.

  For Roundtable, the graph for a single timeline frame looks like:

    ┌──────────┐  ┌───────────┐  ┌────────────┐
    │ Decode   │  │ Decode    │  │ Decode     │  ... (per source)
    │ Frame N  │  │ Frame N-1 │  │ Frame N+1  │
    └────┬─────┘  └─────┬─────┘  └──────┬─────┘
         │              │               │
         ▼              ▼               ▼
    ┌──────────┐  ┌───────────┐  ┌────────────┐
    │ Upload   │  │ Upload    │  │ Upload     │  ... (per source)
    │ CPU→GPU  │  │ CPU→GPU   │  │ CPU→GPU    │
    └────┬─────┘  └─────┬─────┘  └──────┬─────┘
         │              │               │
         └──────────────┴───────────────┘
                              │
                              ▼
                    ┌──────────────────┐
                    │  EffectPass[0]   │  Optional per-layer effects
                    │  (blur, CC, CK)  │  (each effect is a sub-graph)
                    └────────┬─────────┘
                             │
                    ┌──────────────────┐
                    │  EffectPass[1]   │  Chained effects
                    └────────┬─────────┘
                             │
                    ┌──────────────────┐
                    │ TransitionPass   │  A/B track blend
                    │ (dissolve/wipe)  │
                    └────────┬─────────┘
                             │
                    ┌──────────────────┐
                    │ CompositePass    │  N-layer alpha blend
                    │ (final output)   │
                    └────────┬─────────┘
                             │
                    ┌──────────────────┐
                    │  PresentPass     │  → swapchain (playback)
                    │  or ReadbackPass │  → CPU pixels (export/scrub)
                    └──────────────────┘

  Benefits over current monolithic pipeline:
    - Arbitrary layer count (not limited to 32 via composite.comp SSBO)
    - Automatic pipeline barriers → fewer Vulkan validation errors
    - Fault isolation: failed effect shows skip/missing layer, not blank frame
    - Easy to add new pass types
    - GPU timestamp queries at each pass boundary for perf analysis
    - Future: parallel upload + composite of alternating frames

B.2 The Frame Pipeline (Triple-Buffered DAG)
──────────────────────────────────────────────

  FrameClock ticks at frame rate. Each tick produces a RenderGraph for
  the requested frame.  The DAG is built, compiled, and submitted in
  three stages across the triple-buffered ring:

    Slot 0 (recording CPU): Build graph → allocate transient resources
                             → record all passes into cmd buffer
    Slot 1 (submitted GPU):  GPU executing previous frame's DAG
    Slot 2 (fence done):     Resources safe to recycle/evict

  This already works with the existing GpuWorkSubmission triple buffer.
  The change is that instead of one big recording, we build a graph and
  iterate passes.

B.3 Key Design Principle: Layered Abstraction
───────────────────────────────────────────────

  Layer 1 — RenderGraph (new):       Graph data structure + DAG dispatcher
  Layer 2 — RenderPassExecutor (new): Per-pass type executors that call
                                      Compositor, EffectProcessor, etc.
  Layer 3 — CompositeEngine (exists): Orchestrator that builds the graph
                                      from layers[] and submits it
  Layer 4 — CompositeService (exists): Higher-level: layer building, prewarm,
                                       safe mode, calls CompositeEngine

C.1 RenderPass Node Definition
────────────────────────────────

  Each node in the DAG is a render pass:

    enum class PassType : uint8_t {
        External     = 0,  // input from outside (decoded frame, etc.)
        Decode       = 1,  // video decode → CPU buffer
        Upload       = 2,  // CPU buffer → GPU texture
        Effect       = 3,  // texture → shader → texture
        Transition   = 4,  // two textures → blended texture
        Composite    = 5,  // N textures → final output
        Present      = 6,  // final texture → swapchain image
        Readback     = 7,  // GPU texture → CPU staging buffer
        Custom       = 8   // user-defined (future extension)
    };

    struct RenderPass {
        std::string             name;          // debug label
        PassType                type;          // pass category
        uint32_t                passIndex;     // index within the graph (for barriers)

        // Resources: declared as handles into a resource table
        std::vector<ResourceId> inputs;         // textures/buffers consumed
        std::vector<ResourceId> outputs;        // textures/buffers produced

        // Barriers (automatic — can be derived from inputs/outputs usage)
        std::vector<ImageBarrier> preBarriers;  // executed before pass
        std::vector<ImageBarrier> postBarriers; // executed after pass

        // Pipeline state
        VkPipeline              pipeline{VK_NULL_HANDLE};
        VkPipelineLayout        pipelineLayout{VK_NULL_HANDLE};
        std::vector<VkDescriptorSet> descriptorSets;

        // Push constants (if any)
        std::vector<uint8_t>    pushConstants;

        // Shader module references (for pipeline compilation)
        VkShaderModule          shaderModule{VK_NULL_HANDLE};

        // Per-pass error recovery
        bool                    optional{false};  // true = skip on failure
        bool                    fatal{true};      // false = skip, don't crash frame

        // GPU timestamp query
        uint32_t                timestampQueryIndex{UINT32_MAX};
    };

C.2 RenderResource Definition
───────────────────────────────

  Resources are stored in a flat table and referenced by ID:

    using ResourceId = uint32_t;
    constexpr ResourceId kInvalidResource = UINT32_MAX;

    enum class ResourceType : uint8_t {
        Texture,
        Buffer,
        Sampler,
        CombinedImageSampler,
        StorageImage
    };

    enum class ResourceAccess : uint8_t {
        Undefined,
        TransferWrite,
        TransferRead,
        ShaderWrite,
        ShaderRead,
        Present
    };

    struct RenderResource {
        ResourceId              id{kInvalidResource};
        ResourceType            type{ResourceType::Texture};
        std::string             name;

        // Vulkan handle (owned or aliased)
        VkImage                 image{VK_NULL_HANDLE};
        VkImageView             imageView{VK_NULL_HANDLE};
        VkBuffer                buffer{VK_NULL_HANDLE};
        VkSampler               sampler{VK_NULL_HANDLE};
        VkDescriptorImageInfo   descriptor{};

        // Image state tracking
        VkImageLayout           currentLayout{VK_IMAGE_LAYOUT_UNDEFINED};
        ResourceAccess          currentAccess{ResourceAccess::Undefined};
        VkPipelineStageFlags    currentStage{0};

        // Allocation info
        uint32_t                width{0};
        uint32_t                height{0};
        VkFormat                format{VK_FORMAT_R8G8B8A8_UNORM};
        VkImageUsageFlags       usageFlags{0};

        // Pool / lifetime
        bool                    transient{false};  // freed after frame completes
        bool                    external{false};   // owned by caller (swapchain, cache)
        uint32_t                refCount{0};
    };

C.3 RenderGraph Class
───────────────────────

    class RenderGraph {
    public:
        RenderGraph();

        // ── Graph construction ────────────────────────────────────
        ResourceId declareResource(const RenderResource& resource);
        uint32_t   addPass(RenderPass pass);

        // ── Compilation ───────────────────────────────────────────
        // Allocates transient textures/buffers, creates pipelines,
        // assigns descriptor sets, computes automatic barriers.
        bool compile(VkDevice device,
                     VkDescriptorPool pool,
                     const RenderGraphCompileOptions& options = {});

        // ── Execution ─────────────────────────────────────────────
        // Walk passes in topological order, inserting barriers and
        // dispatching each pass.
        bool execute(VkCommandBuffer cmd,
                     uint32_t frameIndex,
                     RenderGraphExecuteOptions* stats = nullptr);

        // ── Resource management ───────────────────────────────────
        void reset();    // clear for next frame
        void destroy(VkDevice device);  // full teardown

        // ── Debug ─────────────────────────────────────────────────
        void dumpGraph(const char* label = nullptr) const;
        [[nodiscard]] size_t passCount() const noexcept;

    private:
        std::vector<RenderPass>     m_passes;
        std::vector<RenderResource> m_resources;
        std::vector<uint32_t>       m_topologicalOrder;  // sorted after compile()
        bool                        m_compiled{false};

        // Internal barrier computation
        void computeBarriers();
        void insertPipelineBarrier(VkCommandBuffer cmd,
                                   const ImageBarrier& barrier);
    };

C.4 Graph Compilation Steps
─────────────────────────────

  1. Topological sort (Kahn's algorithm).  Fail if cycle detected.
  2. For each pass:
     a. If pipeline not already provided, look up or compile from shader
     b. Allocate descriptor sets from pool
     c. Write descriptor set bindings (input/output resources)
     d. Compute automatic barriers from resource access patterns:
        - Track each resource's last access (stage + access mask)
        - Insert preBarrier if next access differs from last
     e. Assign timestamp query index (if queries enabled)
  3. Assign transient resource memory (if using VMA sub-allocation)
  4. Set m_compiled = true

C.5 Graphics Pipeline vs Compute Pipeline
───────────────────────────────────────────

  Roundtable primarily uses compute shaders (VK_PIPELINE_BIND_POINT_COMPUTE)
  for effects, transitions, and compositing.  The render graph should
  support both bind points:

    struct RenderPass {
        // ...
        VkPipelineBindPoint bindPoint{VK_PIPELINE_BIND_POINT_COMPUTE};

        // For graphics passes (spine, text, waveform):
        VkRenderPass         renderPass{VK_NULL_HANDLE};
        VkFramebuffer        framebuffer{VK_NULL_HANDLE};
        VkExtent2D           renderArea{};
        std::vector<VkClearValue> clearValues;
        uint32_t             subpassIndex{0};
    };

  For now, only compute passes are needed.  Graphics passes can be
  added later when the render graph is stable.

D.1 Topological Sort
─────────────────────

  Implement Kahn's algorithm:

    std::vector<uint32_t> topologicalSort(const std::vector<RenderPass>& passes) {
        // 1. Build in-degree map: for each pass, count how many passes
        //    produce resources it consumes.
        // 2. Start with passes that have in-degree 0 (no dependencies).
        // 3. Process queue: for each pass, decrement in-degree of
        //    consumers (passes that read this pass's outputs).
        // 4. If queue empties before all passes processed → cycle detected.
        // 5. Return ordered pass indices.
    }

  Dependency rule: Pass B depends on Pass A if any output resource of
  Pass A is an input resource of Pass B.

D.2 Automatic Barrier Computation
───────────────────────────────────

  Given that barriers are the most error-prone part of Vulkan, the
  render graph should compute them automatically:

    struct ImageBarrier {
        ResourceId          resourceId;
        VkImageLayout       oldLayout;
        VkImageLayout       newLayout;
        VkAccessFlags       srcAccessMask;
        VkAccessFlags       dstAccessMask;
        VkPipelineStageFlags srcStageMask;
        VkPipelineStageFlags dstStageMask;
    };

  Resource access table:

    ResourceAccess         → VkAccessFlags              → VkPipelineStageFlag
    ──────────────────────────────────────────────────────────────────────
    TransferWrite          VK_ACCESS_TRANSFER_WRITE_BIT   VK_PIPELINE_STAGE_TRANSFER_BIT
    TransferRead           VK_ACCESS_TRANSFER_READ_BIT    VK_PIPELINE_STAGE_TRANSFER_BIT
    ShaderWrite            VK_ACCESS_SHADER_WRITE_BIT     VK_PIPELINE_STAGE_COMPUTE_SHADER_BIT
    ShaderRead             VK_ACCESS_SHADER_READ_BIT      VK_PIPELINE_STAGE_COMPUTE_SHADER_BIT
    Present                VK_ACCESS_MEMORY_READ_BIT      VK_PIPELINE_STAGE_BOTTOM_OF_PIPE_BIT

  For each pass, walk the resource usage table:
    1. For each input resource: if currentLayout != expectedLayout,
       insert preBarrier transitioning to expected layout.
    2. For each output resource: if currentLayout != expectedLayout,
       insert postBarrier for next consumer.
    3. Update resource's currentLayout/currentAccess after each pass.

  AIM HIGH: Build a small state tracker so barriers are correct for
  any graph topology, not just linear pipelines.

D.3 DAG Execution Loop
───────────────────────

    bool RenderGraph::execute(VkCommandBuffer cmd, uint32_t frameIndex,
                              RenderGraphExecuteOptions* stats)
    {
        for (uint32_t passIdx : m_topologicalOrder) {
            auto& pass = m_passes[passIdx];

            // 1. Begin timestamp query (if enabled)
            if (pass.timestampQueryIndex != UINT32_MAX && stats) {
                vkCmdWriteTimestamp(cmd,
                    VK_PIPELINE_STAGE_TOP_OF_PIPE_BIT,
                    stats->queryPool, pass.timestampQueryIndex * 2);
            }

            // 2. Insert pre-barriers
            for (const auto& barrier : pass.preBarriers) {
                insertPipelineBarrier(cmd, barrier);
            }

            // 3. Bind pipeline
            vkCmdBindPipeline(cmd, pass.bindPoint, pass.pipeline);

            // 4. Bind descriptor sets
            if (!pass.descriptorSets.empty()) {
                vkCmdBindDescriptorSets(cmd, pass.bindPoint,
                    pass.pipelineLayout, 0,
                    static_cast<uint32_t>(pass.descriptorSets.size()),
                    pass.descriptorSets.data(), 0, nullptr);
            }

            // 5. Push constants
            if (!pass.pushConstants.empty()) {
                vkCmdPushConstants(cmd, pass.pipelineLayout,
                    VK_SHADER_STAGE_COMPUTE_BIT,
                    0, static_cast<uint32_t>(pass.pushConstants.size()),
                    pass.pushConstants.data());
            }

            // 6. Dispatch
            switch (pass.type) {
            case PassType::Upload:
                // Copy buffer to image (vkCmdCopyBufferToImage)
                break;
            case PassType::Effect:
            case PassType::Composite:
                vkCmdDispatch(cmd, groupCountX, groupCountY, 1);
                break;
            case PassType::Readback:
                // vkCmdCopyImageToBuffer
                break;
            // ...
            }

            // 7. End timestamp query
            if (pass.timestampQueryIndex != UINT32_MAX && stats) {
                vkCmdWriteTimestamp(cmd,
                    VK_PIPELINE_STAGE_BOTTOM_OF_PIPE_BIT,
                    stats->queryPool, pass.timestampQueryIndex * 2 + 1);
            }
        }
        return true;
    }

D.4 Pass-Specific Dispatch Helpers
───────────────────────────────────

  Rather than one giant switch, use a dispatch table:

    using PassDispatchFn = bool(*)(VkCommandBuffer, const RenderPass&);

    struct PassTypeDispatch {
        PassType     type;
        PassDispatchFn preprocess;   // called during compile()
        PassDispatchFn execute;      // called during execute()
    };

    // Example: Composite pass dispatch
    bool executeCompositePass(VkCommandBuffer cmd, const RenderPass& pass) {
        // Extract push constants (width, height)
        auto* pc = reinterpret_cast<const CompositePushConstants*>(
            pass.pushConstants.data());

        uint32_t gx = (pc->width  + kCompositeWorkgroupSize - 1) / kCompositeWorkgroupSize;
        uint32_t gy = (pc->height + kCompositeWorkgroupSize - 1) / kCompositeWorkgroupSize;
        vkCmdDispatch(cmd, gx, gy, 1);
        return true;
    }

E.2 Step 2: Build the Graph from Layers[]
───────────────────────────────────────────

    std::shared_ptr<CachedFrame> CompositeEngine::compositeViaRenderGraph(
        ...same params...)
    {
        RenderGraph graph;

        // ── Resource table ────────────────────────────────────────
        // Declare output texture
        ResourceId outputTex = graph.declareResource({
            .name = "compositeOutput",
            .type = ResourceType::StorageImage,
            .width = outW, .height = outH,
            .format = VK_FORMAT_R8G8B8A8_UNORM,
            .usageFlags = VK_IMAGE_USAGE_STORAGE_BIT
                        | VK_IMAGE_USAGE_TRANSFER_SRC_BIT
                        | VK_IMAGE_USAGE_SAMPLED_BIT,
            .transient = true,
        });

        // For each layer:
        std::vector<ResourceId> layerTexIds;
        layerTexIds.reserve(layers.size());

        for (size_t li = 0; li < layers.size(); ++li) {
            // Upload pass
            ResourceId layerTex = graph.declareResource({...});
            uint32_t uploadPass = graph.addPass({
                .name = "uploadLayer" + std::to_string(li),
                .type = PassType::Upload,
                .outputs = {layerTex},
                .optional = true,  // skip on failure
            });

            // Effect passes (if layer has effects)
            ResourceId effectOutput = layerTex;
            for (size_t ei = 0; ei < layers[li].effects.size(); ++ei) {
                ResourceId effectTex = graph.declareResource({...});
                graph.addPass({
                    .name = "effect_" + std::to_string(li) + "_" + std::to_string(ei),
                    .type = PassType::Effect,
                    .inputs = {effectOutput},
                    .outputs = {effectTex},
                    .optional = true,
                });
                effectOutput = effectTex;
            }

            layerTexIds.push_back(effectOutput);
        }

        // Composite pass (merges all layers)
        graph.addPass({
            .name = "finalComposite",
            .type = PassType::Composite,
            .inputs = layerTexIds,
            .outputs = {outputTex},
        });

        // Readback pass (if needed)
        if (!gpuDisplayMode || scrubMode) {
            ResourceId readbackBuf = graph.declareResource({...});
            graph.addPass({
                .name = "readback",
                .type = PassType::Readback,
                .inputs = {outputTex},
                .outputs = {readbackBuf},
            });
        }

        // ── Compile & execute ────────────────────────────────────
        if (!graph.compile(device, descriptorPool)) {
            return nullptr;  // fall back to CPU path
        }

        VkCommandBuffer cmd = m_gpuSubmission->beginRecording();
        if (!graph.execute(cmd, frameIndex)) {
            m_gpuSubmission->endRecording();
            return nullptr;
        }
        m_gpuSubmission->endRecording();
        m_gpuSubmission->submit(computeQueue);
        m_gpuSubmission->waitForCompletion();

        // ── Build result from resources ───────────────────────────
        auto result = std::make_shared<CachedFrame>();
        result->width  = outW;
        result->height = outH;
        // ... extract pixels from readback buffer or image ...
        return result;
    }

E.3 Step 3: Port Upload Pass
──────────────────────────────

  Replace the manual staging ring + vkCmdCopyBufferToImage loop with
  an Upload pass in the graph.  Each layer's upload becomes its own
  graph node.

  Upload pass execution:
    1. Read CPU pixels from CachedFrame or staging buffer
    2. Issue vkCmdCopyBufferToImage into the layer's texture
    3. Transition texture to VK_IMAGE_LAYOUT_SHADER_READ_ONLY_OPTIMAL

  The GpuUploadManager already does this — wrap it as a pass executor.

E.4 Step 4: Port Effect Passes
────────────────────────────────

  Each per-layer effect chain becomes a sequence of Effect passes:

    Layer i, Effect 0 → Effect 1 → ... → Effect N

  Benefits over current code:
    - Each effect is a separate graph node with its own barrier
    - Failed effect (optional=true) shows layer without that effect
    - Easy to add pipeline statistics per effect

  Effect pass execution:
    1. Bind effect shader pipeline
    2. Bind input texture as descriptor set
    3. Bind output storage image as descriptor set
    4. Dispatch compute shader
    5. Transition output to shader-read for next pass

E.5 Step 5: Port Transition Pass
──────────────────────────────────

  The A/B track pair blending currently inlined in CompositeEngine.cpp
  becomes a Transition pass node.

  Transition pass execution:
    1. Bind transition shader pipeline (selected by type)
    2. Bind input A texture, input B texture, output storage image
    3. Push constants: progress, direction, softness
    4. Dispatch compute shader
    5. Output replaces both A and B for downstream passes

E.6 Step 6: Port Composite Pass
─────────────────────────────────

  The final Compositor::composite() call becomes a Composite pass.

  This is the trickiest part because the current compositor uses an SSBO
  with max 32 layers (LayerParamsGPU).  The render graph removes this
  limit by allowing arbitrary layers:

    Option A: Keep 32-layer limit, wrap existing Compositor calls
    Option B: Multi-pass composite — dispatch layers in batches of 32,
              blending intermediates.  This is more flexible but needs
              testing.

  RECOMMENDATION: Start with Option A (wrap existing Compositor as a
  single Composite pass).  Switch to Option B later if >32 layer
  support is needed.

E.7 Step 7: Port Readback Pass
────────────────────────────────

  The current compositor->recordReadback() + mapAndCopyReadback() becomes
  a Readback pass:

    1. vkCmdCopyImageToBuffer (GPU→staging)
    2. Transition output to VK_IMAGE_LAYOUT_GENERAL or PRESENT
    3. After fence signals: vkMapMemory + memcpy to CachedFrame::pixels

E.8 Step 8: Port Present Pass (GPU Display Mode)
───────────────────────────────────────────────────

  For GPU display mode (gpuDisplayMode=true, !scrubMode), the final pass
  presents to the swapchain instead of readback:

    1. Transition composite output to VK_IMAGE_LAYOUT_PRESENT_SRC_KHR
    2. Signal semaphore for swapchain acquisition
    3. No CPU readback

  The VulkanViewport already handles swapchain presentation.  The render
  graph just needs to signal the same semaphore.

E.9 Step 9: Remove Old Monolithic Path
────────────────────────────────────────

  After all passes are ported and tested:
    1. Delete the old monolithic composite() code path
    2. Rename compositeViaRenderGraph() → composite()
    3. Remove the s_useRenderGraph toggle
    4. Clean up now-unused intermediate state in CompositeEngine

F. PHASE D — Fault Isolation & Parallelism
================================================================================

  Estimated effort: 2-3 days

F.1 Per-Pass Error Recovery
─────────────────────────────

  Currently, any failure in upload/effects/transitions/composite returns
  nullptr, triggering CPU fallback for the entire frame.

  With the render graph, each pass has an `optional` flag:

    - UploadPass: optional=true → if upload fails, skip the layer
                  (show it as transparent/empty)
    - EffectPass: optional=true → if effect fails, skip that effect,
                  pass through input texture
    - TransitionPass: optional=true → if transition fails, show a cut
                      (no transition, hard switch at midpoint)
    - CompositePass: optional=false, fatal=true → no output, keep last
                     good frame
    - ReadbackPass:  optional=true → if readback fails, still show
                     GPU frame (just can't export)

  Implementation:
    for (uint32_t passIdx : m_topologicalOrder) {
        auto& pass = m_passes[passIdx];
        bool ok = executePass(cmd, pass);
        if (!ok) {
            if (pass.fatal) {
                return false;  // frame failed
            }
            if (pass.optional) {
                // Skip this pass — resources left in undefined state
                // but downstream passes check isAvailable
                markOutputsUndefined(pass);
                continue;
            }
        }
    }

  AIM HIGH: Instead of returning nullptr for the whole frame, produce
  the best-possible frame with available resources and mark the
  defective layer/pass.  Log which passes failed for debugging.

F.2 Parallel Upload
─────────────────────

  The current pipeline serializes uploads: "upload layer 0, wait, upload
  layer 1, wait, ...".  With the DAG, independent uploads can share the
  same command buffer but be dispatched in any order.

  Further optimization (future): Upload and composite alternating frames
  in parallel across different ring slots:

    Slot 0: Recording frame N (upload + effects + composite)
    Slot 1: GPU executing frame N (composite)
    Slot 2: Decoding frame N+1 (CPU parallel with GPU)

  This requires the FrameProducer/FrameClock architecture to accept
  pipeline overlap — currently out of scope but enabled by the DAG.

F.3 Per-Resource Barrier Granularity
──────────────────────────────────────

  The current code uses a single global VkMemoryBarrier between upload
  and compute stages.  This stalls ALL resources, even if only one layer
  changed.

  With the render graph, barriers are per-resource (VkImageMemoryBarrier):
    - Only barrier textures that actually transition between passes
    - Unchanged textures stay in SHADER_READ_ONLY_OPTIMAL across passes
    - This reduces GPU pipeline stalls

  AIM HIGH: Track per-resource pipeline stage + access mask and emit
  only the barriers that are strictly necessary.

F.4 Layered Effect Sub-Graphs
───────────────────────────────

  When a layer has multiple effects (e.g., Chroma Key → Color Correct →
  Blur), each effect is its own pass node.  The DAG naturally represents
  the chain:

    Layer i effect chain:
      Upload(i) → CK(i) → CC(i) → Blur(i) → (ready for composite)

  If CK fails (optional=true), CC reads from Upload(i) instead.
  If CC fails, Blur reads from CK's output (if CK succeeded) or
  Upload(i) (if CK failed).

  This is superior to the current linked-list approach where a crash
  in any effect breaks the entire pipeline.

  AIM HIGH: Implement a "fallback chain" that tracks the last
  successfully written texture for each layer, so downstream passes
  always have valid input even when upstream passes fail.

G. PHASE E — Unified Resource Management
================================================================================

  Estimated effort: 3 days

G.1 Current Problem
─────────────────────

  Three separate caches with no coordination:

    1. FrameCache (src/core/media/FrameCache.h)
       - Per-source ring buffer of decoded CachedFrame objects
       - Evicts oldest frames when budget exceeded
       - No awareness of GPU texture cache → decoded frames may be
         evicted while GPU still references their textures

    2. GpuTextureCache (src/gpu/GpuTextureCache.h)
       - VRAM LRU of uploaded GPU textures
       - Fence tracking: textures are safe to evict after fence signals
       - No awareness of CPU cache → may re-upload frames that CPU just
         evicted, causing thrash

    3. Composite LRU (inside CompositeEngine)
       - Small ring (8 entries) of composite output frames
       - Used as playback display cache
       - No awareness of either cache above

G.2 UnifiedCache Design
─────────────────────────

    class UnifiedCache {
    public:
        struct Key {
            MediaHandle mediaHandle;
            int64_t     frameNumber;
            ResolutionTier tier;
        };

        struct Entry {
            Key                          key;
            std::shared_ptr<CachedFrame> cpuFrame;    // decoded pixels (RAM)
            GpuTexture                   gpuTexture;  // uploaded (VRAM)
            uint64_t                     lastAccessGeneration;
            uint32_t                     pinCount;     // prevents eviction
            bool                         gpuResident{false};
        };

    private:
        // Budgets (configurable via CacheCoordinator)
        size_t m_cpuBudgetBytes;    // e.g., 4 GB
        size_t m_gpuBudgetBytes;    // e.g., 25% of VRAM

        // Generation counter — incremented each frame
        uint64_t m_generation{0};

        // LRU list (least-recently-used at back)
        std::list<Entry> m_entries;

        // Fast lookup: key → iterator
        struct KeyHash {
            size_t operator()(const Key& k) const;
        };
        std::unordered_map<Key, decltype(m_entries)::iterator, KeyHash> m_index;

        // Eviction policy
        void evictCpu(size_t neededBytes);
        void evictGpu(size_t neededBytes);
        bool canEvict(const Entry& e) const;  // pinCount==0 && !inFlight

    public:
        // ── Access ────────────────────────────────────────────────
        std::shared_ptr<CachedFrame> getCpuFrame(const Key& key);
        GpuTexture getGpuTexture(const Key& key);

        // ── Insert ────────────────────────────────────────────────
        void insertCpu(const Key& key, std::shared_ptr<CachedFrame> frame);
        void insertGpu(const Key& key, GpuTexture texture);

        // ── Pin (prevents eviction while GPU references) ──────────
        void pin(const Key& key);
        void unpin(const Key& key);
        void unpinAll();  // called after fence signal

        // ── Frame lifecycle ───────────────────────────────────────
        void onFrameStart();    // increment generation
        void onFrameComplete(); // check budgets, evict if needed
        void onFenceSignaled(); // clear all pins, safest eviction point

        // ── Shutdown ──────────────────────────────────────────────
        void clear();
    };

G.3 Integration with Render Graph
───────────────────────────────────

  The RenderGraph's resource table aliases into UnifiedCache:

    ResourceId texId = graph.declareResource({
        .name = "upload_media_123_frame_45",
        .type = ResourceType::Texture,
        .external = true,  // owned by cache, not graph
        .image = unifiedCache.getGpuTexture({123, 45, Half}).image,
    });

  When a resource is external, the graph does not manage its lifetime.
  The cache handles eviction based on generation tracking.

G.4 CacheCoordinator Integration
──────────────────────────────────

  The existing CacheCoordinator (referenced in CompositeEngine) should
  be extended to manage the UnifiedCache:

    class CacheCoordinator {
        UnifiedCache* m_unifiedCache{nullptr};

        void onFrameCompleted();  // check VRAM pressure, trigger eviction
        size_t recommendedGpuTexCacheBudget(size_t totalVram);
        void setVramPressureFn(...);
        void onGpuAvailable(size_t vramBytes);
    };

  Or the UnifiedCache can simply replace the CacheCoordinator concept.

G.5 Transient Resource Pool
─────────────────────────────

  In addition to the persistent cache, the render graph needs transient
  textures (intermediate effect targets, transition blend targets, etc.):

    class TransientResourcePool {
        // Pool of VkImage objects keyed by (width, height, format, usage)
        // Reused across frames with fence tracking
        // Freed when all frames referencing them are complete

        struct PoolEntry {
            VkImage     image;
            VkImageView view;
            VkDeviceMemory memory;
            uint64_t    lastUsedGeneration;
            bool        inUse{false};
        };

        VkDevice m_device;
        std::vector<PoolEntry> m_pool;

    public:
        VkImage acquire(uint32_t w, uint32_t h, VkFormat fmt, VkImageUsageFlags usage);
        void release(VkImage image);
        void onFenceSignaled();  // clear inUse flags
        void destroy();
    };

  This prevents allocating and freeing textures every frame, which
  causes VRAM fragmentation and performance spikes.

J. RISK & MITIGATION
================================================================================

  Risk 1: Pipeline barrier correctness
  ──────────────────────────────────────
    Vulkan barriers are notoriously error-prone.  Wrong barriers cause
    rendering corruption or GPU hangs that are hard to debug.

    Mitigation:
      - Enable VK_LAYER_KHRONOS_validation during development
      - Automatic barrier computation from resource access patterns
        (reduces human error)
      - Side-by-side pixel comparison with current pipeline during
        transition period
      - Unit tests for barrier computation (known input → expected output)

  Risk 2: Performance regression
  ────────────────────────────────
    Adding a graph abstraction layer adds CPU overhead.  If the graph
    dispatcher adds >0.5ms CPU time per frame, it negates the benefits.

    Mitigation:
      - Pre-compile graph (compile() is called once, execute() is fast)
      - Keep execution loop tight: no allocations, no branching by
        pass type — use dispatch table
      - Profile CPU overhead with Tracy or similar
      - Benchmark: render graph path must NOT be slower than current
        monolithic path for the same frame

  Risk 3: >32 layer compositing
  ──────────────────────────────
    The current composite shader (composite.comp) uses SSBO with fixed
    32-layer limit.  The render graph doesn't remove this limit by
    itself — it just provides the infrastructure to do multi-pass
    compositing if needed.

    Mitigation:
      - Start with 32-layer limit (same as today)
      - Only implement multi-pass compositing when >32 layers is a
        real requirement (track count shows no NLE project needs more
        than 32 simultaneous video layers in practice)
      - The DAG makes multi-pass compositing TRIVIAL to add later:
        just add another Composite pass that blends the previous
        composite output with the next batch of layers

  Risk 4: Shutdown ordering with new resource management
  ────────────────────────────────────────────────────────
    The UnifiedCache and TransientResourcePool add new objects that
    need proper shutdown ordering alongside GPU context.

    Mitigation:
      - Follow existing ShutdownManager pattern (5-phase ordered teardown)
      - UnifiedCache is destroyed in Phase 4 (DestroyGpu), before
        GpuContext shutdown
      - All transient resources must be released before Phase 4

  Risk 5: Complexity of fault isolation
  ───────────────────────────────────────
    Making every pass resumable on failure adds complexity and testing
    burden.  The "skip and continue" pattern needs careful
    implementation to ensure downstream passes always have valid input.

    Mitigation:
      - Start with fatal-only mode (same as current behavior)
      - Add optional/fatal distinction in a later iteration
      - Use a "last valid output" tracking mechanism:
        each resource remembers its last valid state; if a pass fails,
        downstream passes use the previous valid state

  Risk 6: EngineContracts.h doesn't exist
  ─────────────────────────────────────────
    CompositeGraphExecutor.h references "engine/EngineContracts.h".
    This header may not exist in the codebase yet.

    Mitigation:
      - Before starting Phase A, verify existence of
        engine/EngineContracts.h, RenderGraph, TimelineSnapshot,
        RenderNodeKind, RenderResourceKind
      - If missing, create the minimal header with forward declarations
        and stub types needed by CompositeGraphExecutor
      - Or: absorb the types into the render_graph/ directory directly
