PHASE 1 — Widget Paint Recursion (ROOT CAUSE FIX)
================================================================================

CONTEXT: 197 paintSiblingsRecursive frames found in crash logs. The Qt widget
paint system is entering paint → layout → resize → repaint loops during dock
arrangement at startup and workspace switching. TimelinePanel currently has a
thread_local depth guard that bails at depth > 5, but the ROOT CAUSE — paint
being triggered during widget tree construction when parent/child geometry is
inconsistent — is not addressed. Other widgets (DockBehavior, TimelineTrackWidget,
ProjectPanel, ProgramMonitor) have NO guard.

────────────────────────────────────────────────────────────────────────────────
1.A  Extend paint recursion guards to all deeply-nested widgets
────────────────────────────────────────────────────────────────────────────────

GOAL: Every widget with a custom paintEvent or resizeEvent that participates in
the dock layout tree must have a thread_local recursion guard.

FILES TO MODIFY:

  src/ui/panels/timeline/DockBehavior.cpp
  src/ui/panels/timeline/TimelineTrackWidget.cpp
  src/ui/panels/project/ProjectPanel.cpp
  src/ui/panels/project/ProjectBin.cpp
  src/ui/panels/monitors/ProgramMonitor.cpp
  src/ui/panels/export/ExportPanel.cpp
  src/ui/panels/characters/ShotComposer.cpp
  src/ui/widgets/TimelineRuler.cpp
  src/ui/panels/effects/EffectControlsPanel.cpp

PATTERN TO APPLY (to each file's paintEvent):

    void ClassName::paintEvent(QPaintEvent* event)
    {
        // ── Paint recursion guard ─────────────────────────────────────
        // Detect paint → layout → repaint loops caused by QSplitter + dock
        // widget interactions during startup.  If we re-enter paintEvent
        // more than 5 times without returning, bail out to base class.
        static thread_local int s_paintDepth = 0;
        if (++s_paintDepth > 5) {
            --s_paintDepth;
            QWidget::paintEvent(event);
            return;
        }

        // ... existing paint logic ...

        --s_paintDepth;
    }

PATTERN TO APPLY (to each file's resizeEvent):

    void ClassName::resizeEvent(QResizeEvent* event)
    {
        // Guard against resize → layout → paint → resize recursion during
        // initial dock arrangement.
        static thread_local bool s_inResize = false;
        if (s_inResize) {
            QWidget::resizeEvent(event);
            return;
        }
        s_inResize = true;

        // ... existing resize logic ...

        s_inResize = false;
    }

IMPORTANT: For DockBehavior.cpp, the DockEdgeOverlay::paintEvent (line 362) also
needs this guard. The DockBehavior class has complex layout logic that is a
primary trigger of paint recursion.

────────────────────────────────────────────────────────────────────────────────
1.B  Use setUpdatesEnabled(false) during dock layout construction
────────────────────────────────────────────────────────────────────────────────

GOAL: The root cause of paint recursion is that paint events fire during widget
tree construction when parent/child geometry is inconsistent. Qt's
`setUpdatesEnabled(false)` during setup, followed by `setUpdatesEnabled(true)`
after construction is complete, prevents this entirely. This pattern is already
used in DockLayoutManager.cpp (lines 235-259) and AudioSyncPopulateCards.cpp —
extend it to all panel construction sites.

FILES TO MODIFY:

  src/ui/panels/timeline/TimelineWorkspace.cpp
  src/ui/panels/timeline/TimelineWorkspacePanels.cpp
  src/ui/panels/timeline/TimelinePanel.cpp (rebuildTracks)
  src/ui/MainWindowUI.cpp

CHANGES:

1. In TimelineWorkspace.cpp (constructor or buildPanels method):

    // Wrap dock panel construction in setUpdatesEnabled(false) to prevent
    // paint → layout → repaint recursion during widget tree assembly.
    // This is the same pattern used in DockLayoutManager.cpp.
    setUpdatesEnabled(false);

    // ... existing dock creation, splitter setup, panel wiring ...

    setUpdatesEnabled(true);
    // Force one consolidated layout + repaint after enabling
    updateGeometry();
    repaint();

2. In TimelinePanel::rebuildTracks():

    // Before clearing and rebuilding m_trackWidgets:
    setUpdatesEnabled(false);

    // Clear old track widgets
    for (auto* tw : m_trackWidgets) {
        // ... disconnect signals, remove from layout ...
    }
    m_trackWidgets.clear();

    // Build new track widgets
    for (size_t i = 0; i < trackCount; ++i) {
        // ... create widgets ...
    }

    setUpdatesEnabled(true);
    updateGeometry();

   IMPORTANT: During setUpdatesEnabled(false), any QWidget::update() calls are
   coalesced into a single update on re-enable. This is exactly what we want.

3. In MainWindowUI.cpp (buildPanels or similar large setup method):

    // ... all panel construction ...
    setUpdatesEnabled(true);

PHASE 2 — GPU Pipeline Robustness
================================================================================

CONTEXT: 69 crashes occur primarily in nvoglv64.dll (NVIDIA driver). The root
cause is a combination of:
(a) prewarmPlaybackResources() blocking the UI thread while holding m_compositeMutex
(b) No GPU device-lost recovery — after a TDR, the code continues to call
    mapAndCopyReadback on corrupted state
(c) Tight coupling between audio clock and GPU compositing means any GPU stall
    causes audio glitches and vice versa

────────────────────────────────────────────────────────────────────────────────
2.A  Move prewarmPlaybackResources off the UI thread
────────────────────────────────────────────────────────────────────────────────

GOAL: prewarmPlaybackResources currently runs on the UI thread and holds
m_compositeMutex for up to 400ms. Move it to a std::thread (matching the
existing threading pattern used by FramePresenter, FrameClock, RenderQueue,
DiskFrameWriter, etc.) so the UI thread never blocks.

CRITICAL DESIGN NOTE: The real bottleneck is NOT just that prewarm runs on the
UI thread — it's that it holds m_compositeMutex while polling FrameProducer's
compositor readiness. Holding this mutex blocks FrameProducer from submitting
GPU work, causing TDRs. Moving to a background thread helps (UI stays responsive)
but prewarm still blocks FrameProducer. The real fix is to NOT hold m_compositeMutex
during the sleep-poll loop — use an atomic bool for "is compositor ready" and
only take the mutex for actual data access.

FILES TO MODIFY:

  src/gpu/CompositeService.h
  src/gpu/CompositeServiceFrame.cpp
  src/ui/panels/timeline/TimelineWorkspace.cpp (call site)

CHANGES:

1. In CompositeService.h:
   - Add member: `std::thread m_prewarmThread;`
   - Add member: `std::atomic<bool> m_prewarmRunning{false};`
   - Add member: `std::atomic<bool> m_compositorReady{false};`  ← NEW: lightweight
     flag so FrameProducer's readiness can be checked WITHOUT holding m_compositeMutex
   - The prewarm function signature stays the same (void, tick, outW, outH)

2. In CompositeServiceFrame.cpp, replace prewarmPlaybackResources:

    void CompositeService::prewarmPlaybackResources(int64_t tick, uint32_t outW, uint32_t outH)
    {
        if (!m_timeline || !m_mediaPool || outW == 0 || outH == 0)
            return;

        // Prevent overlapping prewarm runs
        bool expected = false;
        if (!m_prewarmRunning.compare_exchange_strong(expected, true))
            return;

        // Join any previous finished thread
        if (m_prewarmThread.joinable())
            m_prewarmThread.detach();  // detach instead of join to avoid blocking

        // Capture all needed state by value for safety
        // Use weak_from_this() pattern if CompositeService inherits
        // std::enable_shared_from_this; otherwise capture raw pointer with
        // m_destroying guard (Phase 3.A must be done first).
        auto self = shared_from_this();  // requires enable_shared_from_this

        m_prewarmThread = std::thread([this, self, tick, outW, outH]() {
            // ── Phase 3.A guard ──
            if (m_destroying.load()) {
                m_prewarmRunning.store(false);
                return;
            }

            // ── Check compositor readiness WITHOUT holding m_compositeMutex ──
            // Use the lightweight atomic flag instead. When the compositor
            // reports ready, it sets m_compositorReady = true.
            m_compositorReady.store(false);
            {
                // Quick scoped lock JUST to check compositor init
                std::unique_lock lock(m_compositeMutex, std::try_to_lock);
                if (lock.owns_lock()) {
                    auto& gpu = GpuContext::get();
                    if (gpu.isInitialized()) {
                        auto* comp = gpu.compositor(outW, outH);
                        if (comp && comp->isInitialized())
                            m_compositorReady.store(true);
                    }
                }
            }

            if (!m_compositorReady.load()) {
                m_prewarmRunning.store(false);
                return;
            }

            // ── Clip iteration (does NOT need m_compositeMutex) ──
            // Iterate tracks, identify upcoming clips, open media handles.
            // This is read-only on timeline data — if thread safety concerns
            // exist, add a separate m_timelineMutex for the data model.
            // ... existing clip iteration logic (lines ~80-320) ...

            // ── Preroll polling (still needs periodic mutex release) ──
            // Poll loop: check isFrameCached WITHOUT holding m_compositeMutex
            do {
                lastReady = 0;
                for (const auto& t : prerollTargets) {
                    if (m_mediaPool->isFrameCached(t.handle, t.frame, t.tier))
                        ++lastReady;
                }
                if (lastReady == prerollTargets.size()) {
                    allWarm = true;
                    break;
                }
                ++pollCount;
                // Sleep WITHOUT holding any mutex — FrameProducer can work freely
                std::this_thread::sleep_for(std::chrono::milliseconds(2));
            } while (steady_clock::now() < deadline);

            m_prewarmRunning.store(false);
        });
    }

3. At call sites:
   - No change needed — prewarmPlaybackResources() returns immediately.
   - But add the m_destroying guard in callers that could fire during shutdown.

4. SEARCH for all call sites of prewarmPlaybackResources in src/ui/:
   - Verify none of them depend on synchronous completion before calling
     compositeFrame. If any do, restructure with a completion callback.

5. Also set m_compositorReady = true in CompositeService::compositorReady()
   callback (or wherever the compositor signals it's initialized).

NOTE: This requires CompositeService to inherit from
std::enable_shared_from_this<CompositeService>. Add this inheritance if not
already present. The shared_from_this() + m_destroying pattern prevents the
thread from accessing a destroyed CompositeService.

    {
        GpuState expected = GpuState::DeviceLost;
        if (!m_gpuState.compare_exchange_strong(expected, GpuState::Recovering))
            return false;

        spdlog::warn("[GPU] Device lost — attempting recovery");

        // Step 1: Shutdown current GPU state
        shutdown();

        // Step 2: Full re-initialize
        // (Call the same init sequence used at startup)
        bool ok = initialize(/*...*/);  // use same params as original init

        m_gpuState.store(ok ? GpuState::Healthy : GpuState::Failed);

        } else {

────────────────────────────────────────────────────────────────────────────────
2.C  Ensure audio never blocks on GPU (simpler approach)
────────────────────────────────────────────────────────────────────────────────

GOAL: The audio callback runs at 48kHz and is the master clock. Currently,
if the GPU compositor is slow to produce a frame, the audio clock may stall
waiting for it. The fix: ensure the audio buffer NEVER waits for GPU compositing.
Audio writes silence or repeats the last buffer if the next frame isn't ready.
This is a much simpler, lower-risk change than decoupling the entire clock.

FILES TO MODIFY:

  src/core/media/AudioPlaybackService.cpp
  src/core/media/PlaybackController.cpp
  src/core/media/PlaybackController.h

CHANGES:

1. In PlaybackController.h, add:

    // Audio callback safety: if the next audio buffer isn't ready yet,
    // repeat the last buffer instead of blocking. This prevents audio
    // glitches when GPU compositing is slow.
    std::atomic<bool> m_audioNeverBlock{true};

2. In AudioPlaybackService.cpp or the audio callback path, find where it
   waits for the compositor/dispatch and add:

    if (m_audioNeverBlock.load()) {
        // If next buffer not ready, repeat last buffer silently
        // instead of blocking. Audio continues smoothly; video may
        // show a repeated frame, which is far less noticeable than
        // audio crackle or dropout.
        if (!isNextBufferReady()) {
            repeatLastBuffer();
            return;
        }
    }

3. In PlaybackController.cpp, at the start of the composite dispatch path
   that is called by the audio clock tick:

    // Before dispatching: if GPU is overloaded, skip this frame
    // rather than blocking the audio clock. The GPU will catch up.
    if (m_audioNeverBlock.load() && gpuBacklogExceedsThreshold()) {
        // Skip frame dispatch — audio continues on last buffer
        return;
    }

4. This is a MINIMAL change compared to full audio/GPU decoupling. It
   preserves the existing pull-based architecture while adding a safety
   valve for GPU overload scenarios. The user may see a repeated video
   frame instead of hearing audio glitches — a much better trade-off.

PHASE 3 — Systematic Use-After-Free Protection
================================================================================

CONTEXT: 403 ACCESS_VIOLATION crashes — the vast majority are use-after-free
where a background thread, timer callback, or signal/slot delivery references
a widget or object that has been destroyed.

────────────────────────────────────────────────────────────────────────────────
3.A  Add m_destroying + QPointer audit
────────────────────────────────────────────────────────────────────────────────

GOAL: Every class that has background threads, timers, or deferred signal
delivery must have a `std::atomic<bool> m_destroying` flag checked before any
callback accesses member data.

FILES TO MODIFY (scan each and add the pattern):

  src/ui/panels/monitors/ProgramMonitor.h      ✓ already has m_destroying
  src/gpu/CompositeService.h                   needs m_destroying added
  src/core/media/AudioPlaybackService.h        needs m_destroying added
  src/core/media/PlaybackController.h          needs m_destroying added
  src/core/media/MediaPool.h                   needs m_destroying added
  src/ui/panels/timeline/TimelinePanel.h       needs m_destroying added
  src/ui/panels/export/ExportPanel.h           needs m_destroying added
  src/ui/panels/characters/ShotComposer.h      needs m_destroying added
  src/ui/panels/effects/EffectControlsPanel.h  needs m_destroying added
  src/ui/panels/project/ProjectPanel.h         needs m_destroying added
  src/ui/widgets/TimelineTrackWidget.h         needs m_destroying added

PATTERN FOR EACH CLASS:

1. In header, add member:

    #include <atomic>
    // ...
    std::atomic<bool> m_destroying{false};

2. In destructor, SET FIRST before any cleanup:

    ClassName::~ClassName()
    {
        m_destroying.store(true);
        // ... existing destructor logic (stop threads, disconnect signals) ...
    }

3. In every callback/lambda that could fire after destruction starts:

    if (m_destroying.load()) return;

   This includes:
   - Timer callbacks (QTimer::timeout)
   - Thread worker functions
   - Lambda captures passed to QtConcurrent::run
   - Signal/slot connections that could fire during teardown
   - QWidget::showEvent / hideEvent overrides

4. Additionally, audit all QWidget* raw pointer members and convert to QPointer:

   QPointer<QWidget> m_someWidget;  // auto-nulls when widget is destroyed

   FOCUS ON:
   - Any QWidget* stored across async boundaries
   - Any pointer used in lambdas or timer callbacks
   - Any pointer accessed after setCurrentProject or workspace switch

   Example: In TimelinePanel, m_trackWidgets is a vector of raw TimelineTrackWidget*
   pointers. If tracks are rebuilt (rebuildTracks() clears the vector and deletes
   the widgets), any deferred callback holding a raw pointer to an old TrackWidget
   hits use-after-free. There are ~40 iteration sites across TimelinePanel.cpp
   that dereference m_trackWidgets[i] — all of them are vulnerable during
   rebuildTracks() if a paint event or signal fires mid-rebuild.

   FIX Strategy (pick one per case):
   (a) Wrap in QPointer: `std::vector<QPointer<TimelineTrackWidget>> m_trackWidgets`
       — safest but requires null checks at every use site.
   (b) Clear + disconnect before rebuilding: `for (auto* tw : m_trackWidgets)
       tw->disconnect();` THEN delete. This prevents stale signal deliveries.
   (c) Wrap rebuildTracks in setUpdatesEnabled(false) as described in Phase 1.B
       to prevent any paint events from firing during the rebuild window.
   RECOMMENDED: Use (b) + (c) together — disconnect signals + disable updates.

────────────────────────────────────────────────────────────────────────────────
3.B  Define shutdown phase enum and enforce ordered teardown
────────────────────────────────────────────────────────────────────────────────

GOAL: Replace ad-hoc shutdown with a defined phase system so no component
is destroyed while another still references it.

NEW FILE:

  src/core/ShutdownPhases.h

    #pragma once
    #include <atomic>

    namespace rt {

    enum class ShutdownPhase : int {
        Running = 0,        // Normal operation
        Phase1_StopThreads, // Stop background threads (FrameProducer, audio, prefetch)
        Phase2_Disconnect,  // Disconnect Qt signals, null back-pointers
        Phase3_DestroyQt,   // Destroy QWidget tree
        Phase4_DestroyGpu,  // Destroy GPU resources (VkDevice, VMA)
        Phase5_Done         // Complete
    };

    // Global shutdown coordinator
    class ShutdownManager {
    public:
        static ShutdownManager& instance();

        bool isShuttingDown() const {
            return m_phase.load() != ShutdownPhase::Running;
        }
        ShutdownPhase currentPhase() const { return m_phase.load(); }

        // Advance to next phase. Returns false if already past that phase.
        bool advanceTo(ShutdownPhase phase);

    private:
        std::atomic<ShutdownPhase> m_phase{ShutdownPhase::Running};
    };

    } // namespace rt

FILES TO MODIFY:

  src/ui/App.cpp — Enforce phase ordering in ~App()
  src/core/CrashHandler.cpp — Check ShutdownManager before handling crashes
  src/gpu/GpuContext.cpp — Check phase before VMA/VkDestroy
  src/core/media/AudioPlaybackService.cpp — Check phase in audio callback
  src/core/media/MediaPool.cpp — Check phase in worker threads

IMPLEMENTATION:

1. In App.cpp ~App():

    App::~App()
    {
        auto& sm = ShutdownManager::instance();

        // Phase 1: Stop all threads
        sm.advanceTo(ShutdownPhase::Phase1_StopThreads);
        // ... stop audio, stop FrameProducer, stop prefetch workers ...

        // Phase 2: Disconnect all cross-component signals
        sm.advanceTo(ShutdownPhase::Phase2_Disconnect);
        // ... QObject::disconnect, null out raw pointers ...

        // Phase 3: Destroy Qt widgets
        sm.advanceTo(ShutdownPhase::Phase3_DestroyQt);
        // ... delete MainWindow, close all panels ...

        // Phase 4: Destroy GPU
        sm.advanceTo(ShutdownPhase::Phase4_DestroyGpu);
        GpuContext::get().shutdown();

        sm.advanceTo(ShutdownPhase::Phase5_Done);
    }

2. Each background thread callback checks at the top:

    if (ShutdownManager::instance().currentPhase() >= ShutdownPhase::Phase1_StopThreads)
        return;  // or break out of loop

5.A  AudioSync widget pooling instead of destroy+recreate
────────────────────────────────────────────────────────────────────────────────

GOAL: AudioSync's populateCards() destroys and recreates all card widgets
(QFrames, MiniWaveformWidgets, buttons, combos) every time it's called. This
causes visible UI stutter.

FILES TO MODIFY:

  src/ui/panels/audio/AudioSyncCards.cpp
  src/ui/panels/audio/AudioSyncPopulateCards.cpp
  src/ui/panels/audio/AudioSync.h

CHANGES:

1. In AudioSync.h, add widget pool:

    struct CardWidgetPool {
        std::vector<QFrame*> pool;  // pre-allocated, recycled
        size_t activeCount = 0;

        QFrame* acquire();
        void releaseAll();  // hide all, reset activeCount
        void shrink(size_t targetSize);  // delete excess
    };
    CardWidgetPool m_cardPool;

2. In AudioSyncCards.cpp / AudioSyncPopulateCards.cpp:

   Instead of:
        for (auto& match : matches) {
            auto* card = new QFrame(this);
            // ... configure card ...
        }

   Use:
        m_cardPool.releaseAll();  // hide all existing cards
        for (auto& match : matches) {
            auto* card = m_cardPool.acquire();
            if (!card) {
                card = new QFrame(this);
                m_cardPool.pool.push_back(card);
            }
            // ... reconfigure card (set text, reconnect signals) ...
            card->setVisible(true);
        }
        m_cardPool.shrink(matches.size() + 16);  // keep some headroom

3. Implement CardWidgetPool methods:

    QFrame* CardWidgetPool::acquire() {
        if (activeCount < pool.size()) {
            return pool[activeCount++];
        }
        return nullptr;  // caller must allocate
    }

    void CardWidgetPool::releaseAll() {
        for (size_t i = 0; i < pool.size(); ++i) {
            pool[i]->setVisible(false);
        }
        activeCount = 0;
    }

    void CardWidgetPool::shrink(size_t targetSize) {
        while (pool.size() > targetSize) {
            delete pool.back();
            pool.pop_back();
        }
    }

5.C  ProgramMonitor pollTimer optimization (dirty flag approach)
────────────────────────────────────────────────────────────────────────────────

CONTEXT: ProgramMonitor's pollTimer fires every 16ms regardless of whether a
new frame is ready. FramePresenter (in src/core/media/) ALREADY has a proper
thread-based present loop with a wake mechanism via notifyPresenter().
The real issue is that ProgramMonitor's QTimer-based fallback runs continuously.

GOAL: Instead of adding yet another thread (which FramePresenter already provides),
add a simple "dirty flag" check to the existing poll timer so it only does work
when a new frame was actually composed since the last poll.

FILES TO MODIFY:

  src/ui/panels/monitors/ProgramMonitor.cpp
  src/ui/panels/monitors/ProgramMonitor.h

CHANGES:

1. In ProgramMonitor.h, add:

    // Dirty flag: set true whenever a new frame is pushed from the
    // compositor. The pollTimer checks this and skips work if false,
    // saving CPU cycles when the view is static.
    std::atomic<bool> m_newFrameAvailable{false};

2. In ProgramMonitor.cpp, in the method that receives composited frames
   (likely called from FrameProducer or CompositeService):

    void ProgramMonitor::onNewFrame(std::shared_ptr<CachedFrame> frame)
    {
        m_pendingFrame = std::move(frame);
        m_newFrameAvailable.store(true);
        // The pollTimer will pick this up on its next tick
    }

3. In ProgramMonitor::onPollTimer() (around line 498), add at the top:

    // ── Dirty flag optimization ──────────────────────────────────────
    // If no new frame has arrived since the last poll, skip all work.
    // The timer keeps running (to handle window resize, overlay updates,
    // etc.) but the heavy presentFrame path is avoided.
    if (!m_newFrameAvailable.exchange(false) && !m_resizePending) {
        return;  // nothing new to present
    }

4. In presentFrame() or equivalent, at the end:

    // After successfully presenting, the next poll will see
    // m_newFrameAvailable = false and skip until a new frame arrives.

5. Keep the pollTimer at 16ms — this change just makes the callback a
   no-op when there's no work, reducing CPU usage from ~5% to <0.1%.
   The timer interval can be increased to 32ms (30fps) for static views
   once the dirty flag is in place, but that's optional.

NOTE: Do NOT add a condition variable or additional thread here.
FramePresenter already handles the threaded present pipeline.
ProgramMonitor's pollTimer is a UI-thread safety fallback and should
remain lightweight.
