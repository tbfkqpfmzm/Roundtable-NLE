0. LEGAL FRAMING — THE REAL PROBLEM
================================================================================

I am not a lawyer.  Get written confirmation from an IP lawyer before
shipping anything based on this document.

The problem is NOT just linking spine-cpp.  The problem is any chain of
dependency that traces back to the official Spine Runtime implementation:

  Your app
    -> converter tool
        -> official Spine-derived code        ← THIS IS THE DANGER

Even if YOU do not link spine-cpp directly, if your pipeline relies on a
tool that reuses, ports, or derives from the official runtime source code,
the licensing obligation flows through the chain.

  ROUNDTABLE today:
    Downloads .skel + .atlas + .png
    Loads .skel via spine::SkeletonBinary  ← Spine Runtime License applies
    Evaluates via spine::AnimationState    ← Spine Runtime License applies

  ROUNDTABLE after this plan:
    Downloads .skel + .atlas + .png       ← same files, same download
    Parses .skel via rt::SkelBinaryReader ← CLEAN-ROOM, original code
    Evaluates via rt::SkeletonEvaluator   ← CLEAN-ROOM, original code

What a clean-room implementation does:
  ✅ Removes the runtime license obligation (you don't ship their code)
  ✅ Users still need legitimately-created assets (Editor license covers
     the exporter, not the consumer)

What it does NOT do:
  ❌ Eliminate all legal risk — format reverse-engineering without
     authorization can attract legal attention regardless of merit
  ❌ Give you rights to redistribute Spine assets

Honest about .skel:
  The .skel binary format is NOT publicly documented by Esoteric Software.
  Implementing a reader requires reverse-engineering from observed file
  behavior.  In the US, clean-room reverse engineering is generally legal
  for interoperability purposes (Sega v. Accolade, Sony v. Connectix),
  but it carries higher legal attention risk than parsing a documented
  format like JSON.

  The clean-room process (spec writer + separate implementer, Section 1)
  is the legally-recognized method for this.  It is what allowed Compaq
  to clone the IBM PC BIOS and what allowed Connectix to ship the Virtual
  Game Station.

  If you want to avoid reverse-engineering entirely, the JSON export
  format is publicly documented — but your current asset pipeline
  distributes .skel files, not .json.  Switching to JSON delivery
  would require changes on the asset distribution side.

1. WHAT "CLEAN ROOM" ACTUALLY MEANS
================================================================================

For the developers implementing this:

  YOU MUST NOT:
    ❌ Read spine-cpp source code (third_party/spine-cpp/)
    ❌ Read spine-c source code
    ❌ Read the Spine Runtime source in ANY language
    ❌ Port or translate runtime code line-for-line
    ❌ Copy evaluator logic from any runtime
    ❌ Use any third-party converter that derives from Spine runtimes

  YOU MAY:
    ✅ Inspect .skel binary files to observe structure and field layout
    ✅ Read .atlas text files (plain text, self-documenting)
    ✅ Write original parsers, evaluators, and solvers
    ✅ Write original interpolation and hierarchy-solving code

  THE CLEAN-ROOM PROCESS FOR .SKEL DECODING:
    1. One person ("spec writer") reads the spine-cpp SkeletonBinary
       source to understand the binary layout.  They write a purely
       behavioral specification:
         "Offset 0: varint-encoded hash (string table lookup)
          Offset N: varint-encoded bone count
          For each bone:
            Read varint name index → resolve via string table
            Read optional varint parent index (0 if absent = no parent)
            Read float32: x, y, rotation, scaleX, scaleY, shearX, shearY, length
            Read uint8: inherit mode (0=Normal, 1=OnlyTranslation, ...)
          ..."
       The spec must describe BEHAVIOR only — what bytes appear in what
       order, how to interpret them.  No code.  No "how spine-cpp does it."

    2. A DIFFERENT person ("implementer") who has NEVER seen spine-cpp
       source code implements the binary reader from the spec alone.
       If they find gaps in the spec, the spec writer fills them in
       WITHOUT showing code.

    3. The implementer writes unit tests that compare their parser's
       output against known-good data extracted from .skel files by
       a licensed Spine Editor (or by spine-cpp running as a black-box
       reference during development only).

    4. Before release, the implementer signs a statement that they
       never accessed the Spine Runtime source code.

    This is the standard industry practice (Sega v. Accolade, 1992;
    Sony v. Connectix, 2000; Lexmark v. Static Control, 2004).

  │                                            ▼            │
  │                                      .animcache        │
  │                                    (binary on disk)     │
  └─────────────────────────────────────────────────────────┘

  ┌─────────────────────────────────────────────────────────┐
  │                  RUNTIME (every frame)                   │
  │                                                         │
  │  rt::RtSkelPkg ──► rt::SkeletonEvaluator ──► FrameState│
  │                        │                                │
  │                        ▼                                │
  │                   rt::RenderExtractor ──► RenderCmd[]   │
  │                        │                                │
  │                        ▼                                │
  │                   GPU Renderer (Vulkan)                 │
  └─────────────────────────────────────────────────────────┘

Key principle: the import layer does ALL the heavy work (binary parsing,
string deduplication, curve preprocessing).  The runtime layer does
ONLY numeric evaluation — zero strings, zero allocations, zero maps.

No JSON involved in the main pipeline.  JSON import is an optional
secondary path for assets that ship as .json (see Section 3 note).

Four new components (src/core/skeleton/):

  rt::SkelBinaryReader     — parses .skel binary → fills RtSkelPkg
  rt::AtlasParser          — parses .atlas text  → fills RtSkelPkg
  rt::SkeletonEvaluator    — samples animations, solves hierarchy
  rt::RenderExtractor      — produces RenderCommand[] from FrameState

3. IMPORT LAYER (.skel binary parser)
================================================================================

The current pipeline downloads .skel + .atlas + .png.  The importer
reads these directly — no format conversion, no JSON intermediate.

  3A. .SKEL BINARY READER (SkelBinaryReader.cpp)

    The .skel format is a tagged binary stream.  High-level layout
    (determined by behavioral observation, NOT by reading spine-cpp):

      [Header]
        - Version string (e.g. "4.2.0" — null-terminated or length-prefixed)
        - String table (all names used in the skeleton)
        - Bone count + bone data array
        - Slot count + slot data array
        - IK constraint count + IK data
        - Transform constraint count + data
        - Path constraint count + data
        - Skin count + skin data
        - Event count + event data
        - Animation count + animation data

    Common encoding patterns in the binary:
      - Variable-length integers (varint): 7-bit encoding, continuation
        bit in MSB.  Similar to protobuf varint but with Spine-specific
        signed/unsigned variants.
      - Floats: IEEE 754 single-precision, little-endian.
      - Strings: either null-terminated or varint-length-prefixed
        followed by UTF-8 bytes.  Many strings reference the string
        table by index.
      - Booleans: single byte (0 or 1).
      - Colors: four bytes RGBA, or varint-packed uint32.
      - Curves: linear by default.  Stepped: a flag.  Bezier: 4 floats
        (cx1, cy1, cx2, cy2) stored between keyframe values.

    Implementation approach:
      1. Read the version string to determine format variant.
      2. Build the string table (parse once, index by ordinal).
      3. Parse bones, slots in declaration order (parent-before-child
         is guaranteed by the exporter).
      4. Parse skins as slot→attachment mappings.
      5. Parse animations as per-bone, per-slot, per-deform keyframe
         sequences.

    Varint decoding (must be original implementation):
      int32_t readVarint(ByteReader& r) {
          // 7-bit chunks, MSB = continuation
          // Implement from first principles — do NOT port spine-cpp
      }

    Target: parse a typical character .skel file (~500KB) in <50ms.

  3B. ATLAS PARSER (AtlasParser.cpp)

    .atlas files are plain text.  No special library needed.

    Tokenize line-by-line:
      <region_name>.png        → region name
        rotate: true/false     → rotated flag
        xy: X, Y               → pixel offset in atlas page
        size: W, H             → region dimensions
        orig: W, H             → original size (before trimming)
        offset: X, Y           → trim offset
        index: N               → page index

    Compute normalized UVs:
      u0 = xy.x / pageWidth
      v0 = xy.y / pageHeight
      u1 = (xy.x + (rotated ? size.y : size.x)) / pageWidth
      v1 = (xy.y + (rotated ? size.x : size.y)) / pageHeight

  3C. TEXTURE LOADING

    Keep the existing stb_image PNG loading pipeline.
    Textures uploaded to GPU once per character load — unchanged.

  3D. NOTE ON JSON IMPORT (Optional Secondary Path)

    If some assets arrive as .json instead of .skel, a JSON import path
    can be added separately.  The JSON export format is publicly
    documented and carries lower legal risk.  However, this is NOT the
    primary path — your current pipeline distributes .skel files.

    JSON import, if added, would:
      - Parse with nlohmann/json
      - Fill the same rt::RtSkelPkg structures
      - Be gated behind a CMake flag (off by default)
      - Produce identical .animcache files

4. RUNTIME DATA STRUCTURES
================================================================================

All structures in namespace rt.  No "Spine" in any identifier.

  // src/core/skeleton/SkeletonData.h

  struct AtlasRegion {
      int32_t  pageIndex;
      float    u0, v0, u1, v1;   // normalized UV
      uint16_t width, height;
      int16_t  offsetX, offsetY;
      bool     rotated;
  };

  struct BoneData {
      int32_t  parentIndex = -1;    // -1 = root
      float    x = 0, y = 0;
      float    rotation = 0;
      float    scaleX = 1, scaleY = 1;
      float    shearX = 0, shearY = 0;
      float    length = 0;
      uint8_t  inheritMode = 0;     // Normal/OnlyTranslation/NoRotationOrReflection/
                                    // NoScale/NoScaleOrReflection
  };

  struct SlotData {
      int32_t  boneIndex;
      int32_t  defaultAttachmentIndex = -1;   // into AttachmentData[]
      uint32_t color = 0xFFFFFFFFu;           // RGBA packed
      uint8_t  blendMode = 0;                 // Normal/Additive/Multiply/Screen
  };

  enum AttachmentType : uint8_t {
      Region, Mesh, LinkedMesh, BoundingBox, Path, Point, Clipping
  };

  struct AttachmentData {
      AttachmentType type;
      int32_t  regionIndex = -1;         // into AtlasRegion[]
      // Region attachments: 4 corners × (x,y) local positions
      float    localVerts[8];
      float    localUVs[8];
      // Mesh attachments (pools in RtSkelPkg):
      uint32_t vertexOffset = 0;         // into meshVertices[]
      uint32_t vertexCount  = 0;
      uint32_t triangleOffset = 0;       // into meshTriangles[]
      uint32_t triangleCount  = 0;
      uint32_t boneOffset = 0;           // into meshBones[] (skinning)
      uint32_t weightOffset = 0;         // into meshWeights[] (skinning)
      float    width = 0, height = 0;
  };

  struct Skin {
      // (slotIndex, attachmentIndex) pairs sorted by slotIndex
      // for binary search during evaluation
      std::vector<int32_t>  slotIndices;
      std::vector<int32_t>  attachmentIndices;
  };

  // ─── Keyframes ──────────────────────────────────────────
  enum class CurveType : uint8_t { Linear, Stepped, Bezier };

  struct CurveData {
      CurveType type = CurveType::Linear;
      float     bezier[4] = {};      // cx1, cy1, cx2, cy2
      float     lut[100];            // precomputed LUT: u → eased y
  };

  // Parallel arrays for cache-efficient binary search
  template <class V>
  struct Timeline {
      std::vector<float>     times;    // sorted ascending
      std::vector<V>         values;
      std::vector<CurveData> curves;
  };

  struct BoneAnimTrack {
      Timeline<Vector2>   translate;
      Timeline<float>     rotate;
      Timeline<Vector2>   scale;
      Timeline<Vector2>   shear;
  };

  struct SlotAnimTrack {
      Timeline<int32_t>   attachment;   // index into AttachmentData[]
      Timeline<uint32_t>  color;        // RGBA
  };

  struct DrawOrderTrack {
      std::vector<float>            times;
      std::vector<std::vector<int>> orderings;
  };

  struct AnimationClip {
      float                     duration;
      std::vector<BoneAnimTrack>  boneTracks;   // indexed by boneIndex
      std::vector<SlotAnimTrack>  slotTracks;   // indexed by slotIndex
      DrawOrderTrack              drawOrder;
  };

  struct RtSkelPkg {
      // Atlas
      std::vector<AtlasPage>      pages;
      std::vector<AtlasRegion>    regions;

      // Skeleton hierarchy
      std::vector<BoneData>       bones;
      std::vector<SlotData>       slots;

      // Attachments
      std::vector<AttachmentData> attachments;
      std::vector<float>          meshVertices;
      std::vector<uint16_t>       meshTriangles;
      std::vector<int32_t>        meshBones;
      std::vector<float>          meshWeights;

      // Skins
      std::vector<Skin>           skins;
      int32_t                     defaultSkinIndex = 0;

      // Animations
      std::vector<AnimationClip>  animations;
      std::vector<std::string>    animNames;       // string table → index
  };

Design rules:
  • Strings only in BoneData/SlotData names (debug/tools use).
  • All runtime lookups via array index, never by string.
  • No std::map in runtime structures — vectors indexed by boneIndex/slotIndex.
  • No virtual functions.  No dynamic polymorphism.
  • All arrays contiguous for cache efficiency.

5. RUNTIME EVALUATION ENGINE
================================================================================

  struct RuntimeBone {
      float worldX, worldY;
      float a, b, c, d;     // cached 2x2 world matrix for vertex transform
  };

  struct RuntimeSlot {
      int32_t  attachmentIndex;   // -1 = none
      uint32_t color;
  };

  struct FrameState {
      std::vector<RuntimeBone>  bones;     // size = bone count
      std::vector<RuntimeSlot>  slots;     // size = slot count
      std::vector<int32_t>      drawOrder; // size = slot count
  };

  void evaluateFrame(
      const RtSkelPkg&        pkg,
      int32_t                 animIndex,
      int32_t                 skinIndex,
      float                   time,          // wraps for looping
      FrameState&             out);          // reused every frame

  Pipeline (single-threaded, no allocations):

    1. SAMPLE BONE TRACKS
       for each bone B:
         local[B] = composeMatrix(
             boneData[B],
             translate[B].sample(time),
             rotate[B].sample(time),
             scale[B].sample(time),
             shear[B].sample(time))

    2. SOLVE HIERARCHY (single linear pass, parent before child)
       for each bone B in array order:
         if parentIndex >= 0:
             world[B] = world[parentIndex] * local[B]
         else:
             world[B] = local[B]
         // Cache 2x2 for vertex transform:
         out.bones[B] = { worldX, worldY, a, b, c, d }

    3. RESOLVE SKIN (attachment overrides)
       for each entry in skin:
         out.slots[slotIndex].attachmentIndex = skinAttachment

    4. SAMPLE SLOT TRACKS (attachment swaps, color)
       for each slot S:
         if slotTracks[S].attachment has keys:
             out.slots[S].attachmentIndex = sample
         if slotTracks[S].color has keys:
             out.slots[S].color = sample

    5. RESOLVE DRAW ORDER
       if drawOrderTrack has keyframe at time:
           out.drawOrder = interpolated ordering
       else:
           out.drawOrder = [0, 1, 2, ..., slotCount-1]

  KEYFRAME SAMPLING:
    template <typename V>
    V sampleTimeline(const Timeline<V>& tl, float time) {
        // Binary search with std::lower_bound
        // Linear:   lerp(val0, val1, t)
        // Stepped:  val0
        // Bezier:   lerp(val0, val1, sampleLUT(curve.lut, t))
    }

  BEZIER EVALUATION:
    At import time, precompute 100-sample LUT per Bezier curve:
      for i in 0..99:
          u = i / 99.0
          solve x(t) = u for t (Newton, 8 iterations at import time)
          lut[i] = y(t)
    At runtime: sample LUT with linear interpolation — 2 lookups + 1 lerp.
    No Newton iteration at runtime. No per-frame Bezier solving.

  BONE INHERITANCE (during hierarchy solve):
    Normal:                 world = parent * local
    OnlyTranslation:        inherit position only
    NoRotationOrReflection: inherit position + scale, ignore rotation
    NoScale:                inherit position + rotation, ignore scale
    NoScaleOrReflection:    inherit position + rotation, no scale

6. RENDER COMMAND GENERATION
================================================================================

  struct RenderCommand {
      int32_t  texturePageIndex;
      uint32_t firstVertex;
      uint32_t vertexCount;
      uint32_t firstIndex;
      uint32_t indexCount;
      uint8_t  blendMode;
  };

  void extractRenderCommands(
      const RtSkelPkg&                pkg,
      const FrameState&              state,
      std::vector<RenderCommand>&    outCommands,
      std::vector<Vertex>&           outVertices,   // pre-allocated, cleared
      std::vector<uint16_t>&         outIndices);

  For each slot in draw order (front to back):
    1. Get attachment from state.slots[slotIndex].attachmentIndex
    2. If none → skip
    3. Get bone world matrix from state.bones[boneIndex]
    4. Transform attachment vertices by the world matrix
    5. Emit vertices + indices into output buffers
    6. Emit RenderCommand with texture page, blend mode, buffer ranges

  The GPU renderer receives RenderCommand[] and processes identically
  to today's SpineRenderBatch[] — same Vulkan pipeline, same draw calls,
  same blend states, same texture binding.  Only the vertex source changes.

7. BINARY ANIMATION CACHE (.animcache)
================================================================================

Parsing .skel on every character load is wasteful.  Import once, cache
in a fast binary format.

  Per-character cache file:
    assets/Converted/Binary/<CharName>/<outfit>/skeleton.animcache

  File format:
    [Header]
      uint32_t magic       = 0x4B4C5352  // "RSKL"
      uint32_t version     = 1
      uint32_t boneCount, slotCount, regionCount
      uint32_t attachmentCount, skinCount, animCount
      uint32_t stringTableSize

    [BoneData array]          boneCount × BoneData (binary dump)
    [SlotData array]          slotCount × SlotData
    [AtlasRegion array]       regionCount × AtlasRegion
    [AttachmentData array]    attachmentCount × AttachmentData
    [Mesh vertex pool]        raw floats
    [Mesh triangle pool]      raw uint16_t
    [Mesh bone pool]          raw int32_t
    [Mesh weight pool]        raw floats
    [Skin array]              skinCount × (slotIndices + attachmentIndices)
    [Animation array]         animCount × AnimationClip (with timelines)
    [String table]            concatenated null-terminated strings

  Loading: mmap or fread the entire file.  Fix up internal offsets.
  Load time: <50ms (vs. ~200ms for .skel parse).

  Cache invalidation: store source .skel mod time + size in header.
  If source is newer, re-import.

    Feature                 Priority
    ─────────────────────────────────────────

    Mesh deform (FFD)       REQUIRED (per-vertex anim)

    Feature                 Priority
    ─────────────────────────────────────────
    IK constraints          LATER
    Transform constraints   LATER
    Path constraints        LATER

  Never needed: physics, bounding boxes, path attachments, point attachments

11. CLEAN-ROOM RULES FOR DEVELOPERS
================================================================================

These rules are NOT optional.  Violating them recreates the problem.

  RULE 1 — Implementer NEVER reads spine-cpp source
    The person writing SkelBinaryReader, SkeletonEvaluator, and
    RenderExtractor MUST NOT look at third_party/spine-cpp/.

  RULE 2 — Spec writer writes behavioral spec only
    The spec writer MAY read spine-cpp SkeletonBinary.cpp to understand
    the binary layout.  Output must be pure behavioral description
    (field order, types, encodings) — NO code, NO pseudocode,
    NO "spine-cpp does it like this."

  RULE 3 — Implementer uses spec only
    The implementer works from the behavioral spec + observed .skel
    file structure.  If the spec is incomplete, the spec writer fills
    gaps without showing code.

  RULE 4 — Unit tests are the gate
    Tests compare output against spine-cpp's evaluation.  The test
    harness may USE spine-cpp as a black-box oracle (it's test
    infrastructure, not shipped code).  The implementer does not
    read the oracle's internals.

  RULE 5 — Import only; never runtime-parse
    Runtime loads .animcache — never .skel, .atlas, or .json.

  RULE 6 — Naming discipline
    No "Spine" in any identifier, filename, or namespace.
    Use rt::Skeleton*, rt::Character*, rt::Animation*, etc.

  ✗ Does not give rights to redistribute Spine-format assets.

13. ALTERNATIVES
================================================================================

  If format reverse-engineering risk is too high:

    Option A — Switch asset distribution to JSON
      Have your asset pipeline distribute .json exports instead of
      .skel.  JSON is publicly documented — lower legal risk.
      Implementation: JSON parser (nlohmann/json) replaces the
      SkelBinaryReader.  All other components unchanged.

    Option B — Switch to a different format entirely
      The runtime evaluator + renderer work on rt::RtSkelPkg.
      Only the importer knows the source format.
      Alternative importers:
        • DragonBones (.dbbin/.json) — MIT license
        • glTF 2.0 skinned mesh — ISO standard
        • Custom in-house format — zero legal risk

    Option C — Negotiate with Esoteric
      Contact Esoteric Software about a commercial license that
      covers your use case without the per-user clause.
