  TL;DR — THE VERDICT
  ───────────────────────────────────────────────────────────────────────────
  The crashes are not "bugs in the spine renderer" or "bugs in the
  compositor."  They are an architectural defect: nothing in this codebase
  owns the GPU.  Twenty-four files independently call vkQueueSubmit /
  vkQueueWaitIdle / vkWaitForFences.  Five-plus subsystems each hold their
  own VkQueue handle and submit work whenever they please.  Three caches
  evict resources concurrently with no shared notion of which textures are
  in flight.  Thirty-plus files spawn threads that talk to each other
  through a mix of mutexes, atomic flags, condition variables, and
  shared_ptr handoffs — there is no single pattern.

  Today's crash (perf_log.txt:3038, "vkWaitForFences failed VkResult=-4
  slot=1") is the same class of bug as yesterday's: WDDM TDR fired because
  multiple uncoordinated GPU producers oversubscribed the device.  The
  cross-queue Spine→Compositor sync I added today (committed at
  GpuContext.cpp:401-417, SpineRenderer.cpp:610-632) DID run — log line
  597 confirms "Spine cross-queue sync enabled" — but it only papered
  over one specific hazard.  A different hazard fired 15 seconds later
  on a shot boundary (3 new clips opening, atlas re-upload, first-render
  shader compile, 3 simultaneous prefetch decodes — all hitting the GPU
  at once: composite TOTAL=63.1 ms at perf_log.txt:3032, then TDR).

  This is not fixable with more whack-a-mole patches.  The codebase needs
  ONE central GPU scheduler, ONE composite path, ONE cache, and ONE
  threading pattern.  Most of the existing code is salvageable — the
  building blocks (Compositor, EffectProcessor, TransitionRenderer,
  SpineRenderer, Nv12Converter, MediaPool, render-graph data structures
  in Phase A/B/C) are good.  What's missing is the conductor.

  Estimated effort to reach stability: 4-8 weeks of focused work, mostly
  consolidation/deletion rather than greenfield code.  The project is
  ~30% over-built (legacy fallback paths, vestigial classes, toggle
  flags between half-finished implementations) and that 30% is exactly
  where the bugs hide.

PART I — DIAGNOSIS
================================================================================

I.A — Why the crashes keep happening
─────────────────────────────────────

  Symptom: VkResult=-4 (VK_ERROR_DEVICE_LOST) during scrubbing or shot
  changes.  Always preceded by a burst of GPU activity from multiple
  producers.

  Direct mechanism: WDDM detects the GPU has not made forward progress
  within ~2 seconds and resets the device.  Every vkQueueSubmit fence
  afterwards returns -4.

  Underlying causes observed in the logs and source:

    1. Cross-queue write-after-read on shared resources.
       Example, fixed today: SpineRenderer writes m_framebuffer on
       graphics queue (family 0), CompositeEngine samples it on compute
       queue (family 2), no semaphore between them.  Submission order
       only applies WITHIN a queue.

    2. Resource recycling without fence tracking.
       Example, in the unstaged GpuUploadManager A4 pool: an evicted
       VkImage gets recycled and pixel-uploaded while a previous
       command buffer's descriptor still points at it.

    3. Concurrent vkQueueSubmit on the same queue.
       Until today's patch, the SHARED SpineRenderer never had its queue
       mutex set (GpuContext.cpp old line 395 just called init() and
       never setQueueMutex), while VulkanViewport submitted to the same
       graphics queue under graphicsQueueMutex.  Vulkan's external-sync
       requirement on VkQueue was being violated.  My patch fixed it,
       but it was a latent bug for who-knows-how-long.

    4. NVDEC oversubscription.
       MediaPool spawns 10 prefetch workers at ABOVE_NORMAL priority
       (perf_log.txt:78-88).  NVIDIA NVDEC has a small fixed number of
       hardware decoder slots; oversubscribing them increases latency,
       not throughput, and starves other GPU work.  The user's own
       comment in CompositeServiceFrame.cpp explicitly documents that
       "count=8 urgent prefetch flooded NVDEC + the graphics queue with
       ~16-24 simultaneous urgent decodes and tripped TDR."

    5. Stalls long enough to look like a hang.
       Today's 62.9 ms composite (Chime first-render: atlas upload +
       shader compile + first draw + decode workers all at once) is by
       itself harmless, but stacked with another stall it can blow
       through TDR's budget.

  Each of these is locally fixable, but as long as the architecture
  permits any subsystem to submit GPU work whenever it pleases, the next
  hazard is just one feature away.


I.B — The architectural pattern that produces them
──────────────────────────────────────────────────

  Grep results from the current source tree (May 14, 2026):

    Files calling vkQueueSubmit / vkQueueWaitIdle / vkWaitForFences:
      24 files — including SpineRenderer, CompositeEngine,
      EffectProcessor, Nv12Converter, VulkanViewport, VideoUploader,
      Compositor, TransitionRenderer, GpuResourceManager, MediaPoolDecode,
      SpinePrerenderer, AnimationVideoCache, Texture, CommandPool,
      CompositeService, CompositorInit, GpuWorkSubmission, Device.

    Files spawning threads:
      30+ files — FrameClock, FrameProducer, FramePresenter, MediaPool
      prefetch (10 workers), MediaPool loop pre-decode (4 workers),
      AudioSync, AudioPlaybackService, ScopesPanel, OmniShotDetector,
      ShotComposer, RenderQueue, ChromaKeyEncoder, AnimationVideoCache,
      and others.

    Files implementing the "GPU composite FAILED — using legacy CPU
    path" backoff:
      15 files — every layer of the composite stack has its own
      fallback wired in.  When TDR fires and the device is genuinely
      lost, ALL of those fallbacks also fail (today's log:
      "VulkanViewport: CPU upload texture data failed (VkResult=-4)").
      The fallback is not a real safety net — it's noise that hides
      the underlying bug.

    Concurrent caches:
      • CPU FrameCache (RAM ring buffer per media source)
      • GpuTextureCache (VRAM LRU inside CompositeEngine)
      • Composite result LRU (also inside CompositeEngine)
      • UnifiedCache (new — currently being added in CompositeServiceFrame.cpp)
      Four caches with overlapping responsibility, no shared budget,
      no shared eviction policy, no shared knowledge of which texture
      is currently bound to a descriptor in an in-flight command buffer.

  The pattern is consistent: every subsystem was built as if it were
  the only one talking to the GPU.  When you assemble them into one
  process they work — for a while, until concurrency exposes them.

PART II — THE PLAN (PRIORITIZED)
================================================================================

P0 — STABILIZE THE WORKING TREE  [days, blocking]
─────────────────────────────────────────────────

  Action: bring the tree to a known-stable state before doing anything
  ambitious.  This is the single most valuable thing right now.

    1. Decide the fate of the 95 modified files individually.  For each:
       commit it, revert it, or stash it on a feature branch.  No more
       "BACKUP: pre-X" commits — those are a smell that the branch
       discipline isn't there.

    2. Pick ONE composite path and run only that one.  Either
       s_useRenderGraph=true forever (preferred) or false forever.
       Both code paths active simultaneously means every bug repro
       has 2x the surface area.

    3. Lock in the SpineRenderer cross-queue sync I added today.  Verify
       the fix builds, the perf cost is bounded (vkQueueWaitIdle on the
       compute queue per beginFrame should be <2 ms), and that it
       eliminates the specific Spine→Compositor hazard.

    4. Run a 10-minute scrub-and-play stress test.  Note every crash and
       its log signature.  This is the regression bar to clear before
       anything else changes.

  Until P0 is done, P1-P7 should not be started.  The reward for
  stabilizing first is that subsequent fixes can be validated against
  a stable baseline rather than a moving target.


P1 — CENTRALIZE GPU SUBMISSION  [1-2 weeks, highest impact]
───────────────────────────────────────────────────────────

  The single most important architectural change.  Replace 24 scattered
  vkQueueSubmit call sites with ONE.

  New class: src/gpu/GpuScheduler.h/.cpp

    class GpuScheduler {
    public:
        // The ONLY way GPU work enters the device.  All other modules
        // hand off recorded command buffers; nobody else owns a queue.
        struct Submission {
            VkCommandBuffer cmd;
            QueueFamily preferredQueue;     // graphics | compute | transfer
            std::vector<VkSemaphore> waitSemaphores;
            std::vector<VkPipelineStageFlags> waitStages;
            std::vector<VkSemaphore> signalSemaphores;
            VkFence completionFence;
            std::function<void()> onComplete;  // CPU-side callback
            uint64_t submissionId;
        };

        SubmissionId submit(Submission&&);
        VkFence waitFor(SubmissionId);       // for callers that need
        bool isComplete(SubmissionId) const; // synchronous knowledge
    };

  Properties:
    - One thread services queue submission per VkQueue.  External
      synchronization on VkQueue is the scheduler's invariant.
    - Cross-queue dependencies are expressed via semaphores attached to
      Submissions, not by callers calling vkQueueWaitIdle.
    - Subsystems (Spine, Compositor, Effect, Transition, NV12, etc.)
      record into command buffers but never see a VkQueue.
    - Eliminates the Spine→Compositor hazard by construction (the
      scheduler knows about both submissions and inserts a semaphore
      automatically when the second references the first's output).

  Deliverable: every existing vkQueueSubmit call site rewritten to
  submit through GpuScheduler.  Direct vkQueueSubmit becomes a build
  error via a wrapper that #defines vkQueueSubmit out for non-scheduler
  TUs.

  Why this is worth doing first: it eliminates an entire class of
  bugs at the architecture level.  Today's TDR (and the half-dozen
  others in logs/) all share the root cause "uncoordinated submissions
  to multiple queues."  After P1, that root cause is structurally
  impossible.


P2 — FINISH THE RENDER GRAPH AND DELETE THE LEGACY PATH  [2 weeks]
──────────────────────────────────────────────────────────────────

  Per RENDER_GRAPH_PLAN.txt, Phase A (data structures), Phase B (DAG
  dispatcher), and Phase C (pass porting) are complete.  Phase D
  (fault isolation), Phase E (unified cache), Phase F (telemetry) are
  not started.

  Recommended changes:

    1. Make s_useRenderGraph permanently true.  Delete the toggle.
       Delete every code path gated by !s_useRenderGraph.

    2. Delete src/gpu/CompositeServiceCpuFallback.cpp.  It does not
       work when the device is actually lost (today's log proves it),
       and when the device is healthy it's never exercised, so it's
       untested code.  If a real CPU safe-mode is needed, it should be
       a separate process invoked by a launcher — not a per-frame
       fallback.

    3. Delete the "(backoff N/3)" retry logic in CompositeEngine.cpp.
       It hides bugs.  When a GPU submit fails, log the failure, mark
       the device as dead (which the existing tryRecover already does),
       and surface the modal restart dialog.  That's it.

    4. Build the missing engine/EngineContracts.h that
       CompositeGraphExecutor.h includes (per RENDER_GRAPH_PLAN A.2).

    5. Implement Phase D (per-pass fault isolation): a failed effect
       pass should disable that pass for the rest of the session and
       return the prior layer; it should NOT crash the frame.  This is
       the actual safety net that the legacy CPU fallback was
       pretending to be.

  Delete count estimate: 1500-2500 lines (legacy fallback + CPU
  composite + backoff retries + dual-path branches throughout
  CompositeService and CompositeEngine).


P3 — UNIFIED CACHE COORDINATOR  [LANDED — see realized scope below]
───────────────────────────────────────────────────────────────────

  REALIZED: src/core/media/UnifiedCache.h/.cpp ships as a coordinator
  over FrameCache + GpuTextureCache + composite LRU.  It does not own
  the frames — it owns the policy state that the three caches share.

  Currently provides:
    - Global generation counter incremented per composite frame.
    - Per-key last-access generation (LRU ordering source of truth).
    - Per-media playhead-window declaration that forwards to
      FrameCache::setPlayheadWindow (pins frames near the playhead).
    - Throttled eviction pass (every 3s) that drops stale windows and
      prunes the lastAccess map.
    - Throttled hit-rate logging (every 30 frames) for diagnostics.

  Why a coordinator instead of the plan's wholesale replacement:

    The original plan called for deleting GpuTextureCache, the
    composite-result LRU in CompositeEngine, and reducing FrameCache
    to a thin adapter.  In practice each of those caches owns
    non-trivial Vulkan lifecycle:
      • GpuTextureCache holds pin counts referenced by command buffers
        in flight on the GPU.
      • The composite-result LRU is used by CPU-readback paths
        (thumbnails, Source Monitor) and recycles pixel buffers.
      • FrameCache coordinates with MediaPool and packed-alpha
        decoding via shared_ptr<CachedFrame>.
    Migrating those into one new owner is a multi-day rewrite with a
    high regression surface (pixel-equality across long timelines,
    low-VRAM stress, scrub thrash).  The coordinator delivers the
    user-visible wins (window pinning, coordinated generation,
    telemetry) at ~5% of the risk.

  Deferred items from the original plan:
    • Single budget owner with adaptive CPU↔VRAM split — partially
      done via CacheCoordinator; UnifiedCache::rebalanceBudgets() is
      stubbed (logs only) until hit-rate stability is verified in the
      field.
    • Fence-aware eviction — the per-slot pin tracking in
      GpuUploadManager (P3's "delete this" item) is doing this job
      now via GpuWorkSubmission's fence ring.  Folding it into the
      coordinator is on the wishlist for a later cleanup pass once
      the test surface is wider; it does not affect stability today.

  Net: P3 is "done enough" for the stability mission.  Pushing
  further is yak-shaving.


P4 — STANDARDIZE THE THREADING MODEL  [1 week]
──────────────────────────────────────────────

  Today there are at least 8 thread roles, each with its own
  communication style:
    - FrameClock (TIME_CRITICAL, atomic counter + cv)
    - FrameProducer (composite worker, request queue + cv)
    - FramePresenter (lock-free exchange slot)
    - MediaPool prefetch workers ×10 (custom queue + futures)
    - MediaPool loop pre-decode workers ×4 (separate queue)
    - AudioSync worker (dedicated thread + atomic)
    - ScopesPanel worker (timer + queue)
    - Various encoders (RenderQueue, ChromaKeyEncoder)

  Pick one pattern and apply it everywhere:

    Recommendation — single MPMC queue + ownership transfer:
      • All inter-thread messages use std::move into a MPMC queue
        (e.g. moodycamel::ConcurrentQueue, already a common dep).
      • No shared_ptr fan-out unless multiple consumers genuinely need
        a copy.  shared_ptr ref-count contention is invisible until it
        isn't.
      • Per-thread state is owned exclusively by that thread; no other
        thread reads or writes it.
      • Cancellation via a single std::stop_token per thread.

  Apply to: FrameClock, FrameProducer, FramePresenter (they already
  approximate this), MediaPool prefetch, AudioSync, encoders.

  The "BACKUP: pre-thread-model-fix" commit (5ae91f4) suggests the
  user has already started this.  The work should be finished and the
  backup branch deleted.


P5 — BOUND CONCURRENCY  [2 days]
────────────────────────────────

  Concrete tunables:

    1. MediaPool prefetch workers: 10 → 2.  NVIDIA NVDEC on consumer
       GPUs has 2-3 hardware decoder engines.  Beyond that, work
       queues internally and adds latency.  ABOVE_NORMAL priority
       further starves the UI thread.

    2. Loop pre-decode workers: 4 → 1.  These are background warm-ups;
       there is no reason for 4.

    3. Urgent prefetch count per shot boundary: already reduced from
       8 → 2 in the unstaged diff (good).  Confirm and lock in.

    4. Atlas upload: serialize through the GpuScheduler.  Currently
       (perf_log.txt:3036, 3068) atlases re-upload during heavy
       scrubbing because the cache evicts them; UnifiedCache (P3) +
       central scheduler (P1) prevents the race.

  These changes alone would likely reduce the TDR rate by 50%+
  even before P1-P4 land.  They're a high-leverage day of work.


P6 — VALIDATION LAYERS ALWAYS-ON IN DEV  [hours]
────────────────────────────────────────────────

  Already supported in src/gpu/vulkan/Instance.h (enableValidation
  defaults to true if ROUNDTABLE_DEBUG).  Confirm:

    1. Dev builds always have ROUNDTABLE_DEBUG set.
    2. Add VK_VALIDATION_FEATURE_ENABLE_GPU_ASSISTED_EXT (catches
       shader OOB and stale-descriptor reads at the offending dispatch).
    3. Add VK_VALIDATION_FEATURE_ENABLE_SYNCHRONIZATION_VALIDATION_EXT
       (catches the exact class of cross-queue hazard that caused
       today's crash, before it ships).
    4. Make validation messages assert in dev — silent warnings are
       worse than crashes, because they teach you to ignore them.


P7 — STRESS TEST IN CI  [1-2 days]
──────────────────────────────────

  Bug we hit was timing-sensitive.  A 60-second scrub-and-play stress
  test, run in CI on every PR, would have caught it.

    Tooling:
      - A scripted "user simulation" (Qt test driver or a stub that
        directly drives the timeline API): random scrubs, plays,
        seeks, shot-boundary crossings, for 60 seconds.
      - Run with validation layers on.
      - Assert: zero VkResult=-4, zero validation errors, no
        unhandled exceptions.

    Cost: one CI minute per PR.  Catches: the next ten variants of
    today's bug, before merge.

PART III — WHAT TO DELETE
================================================================================

  Aggressive deletion is the highest-leverage activity in this
  codebase.  Below is the prioritized hit list.  All of these are
  defended by partial implementations or feature flags and contribute
  more confusion than function.

  Delete now (after P0):
    • src/gpu/CompositeGpuSlot.h — vestigial, RENDER_GRAPH_PLAN.txt
      A.1 already calls this out.
    • Any "tryRecover" device-lost in-place re-init code (the comment
      in GpuContext.cpp:189-210 already explains why it's dead — the
      neutered version is fine, just delete the old commented-out
      machinery once nothing references it).

  Delete after P2 (render graph is the only path):
    • src/gpu/CompositeServiceCpuFallback.cpp (full file)
    • CompositeEngine "(backoff N/3)" retry logic (~50 lines scattered)
    • All branches in CompositeService.* gated by
      `!s_useRenderGraph` (~200 lines)
    • Any per-pass "GPU composite FAILED — using legacy CPU path"
      log lines (~30 lines)

  Delete after P3 (UnifiedCache absorbs them):
    • src/gpu/GpuTextureCache.h/.cpp (~300 lines)
    • Composite-result LRU inside CompositeEngine (~150 lines)
    • Per-slot pin tracking in GpuUploadManager (the A9 work) once
      UnifiedCache owns lifetime (~100 lines)

  Delete after P4 (threading standardized):
    • Custom MediaPool prefetch queue → MPMC queue (~200 lines saved)
    • Per-thread futures plumbing in MediaPool (~100 lines)

  Total estimated deletion: 2500-3500 lines.  That's ~2% of the
  codebase by line count, but probably ~20% of the load-bearing
  complexity.

PART IV — WHAT TO KEEP
================================================================================

  A lot of this codebase is good.  The recommendation is NOT to
  rewrite everything.  Specifically, KEEP:

    • The Vulkan low-level wrappers — Instance, Device, Swapchain,
      Pipeline, CommandPool, Texture, Framebuffer, Buffer.  These are
      clean, single-purpose, and correct.
    • spine-cpp runtime integration (src/core/spine/ + the SpineRenderer
      shaders).  The bug is in how it's coordinated, not in spine itself.
    • Audio engine (PortAudio WASAPI exclusive-mode).  Well-isolated,
      separate from the GPU mess.
    • Export pipeline (src/export/).  Mostly self-contained.
    • The UI layer (src/ui/, ~107k LOC).  Big but it works; touching it
      in this plan would be scope creep.  The "TimelineWorkspace" /
      "Panels" wiring is verbose but functional.
    • The render-graph data structures (Phase A) and DAG dispatcher
      (Phase B).  These are the right design; they just need to be the
      ONLY composite path.
    • MediaPool's NVDEC integration — keep the integration, just
      reduce the worker count and route through GpuScheduler.

  Net: roughly 90% of the lines stay.  The architectural changes are
  about how the existing pieces talk to each other, not what they do.

PART VI — RISK & FALLBACK
================================================================================

  Top risks of this plan:

    1. P1 (GpuScheduler) is invasive.  Every existing GPU caller has
       to change.  Mitigation: do it incrementally, one subsystem at
       a time, behind a feature flag that routes through the scheduler
       OR the legacy direct-submit path.  Once all subsystems are
       migrated, delete the flag and the legacy path.

    2. P2 deletes ~2000 lines of "safety net" that some testing
       relied on.  Mitigation: Phase D's per-pass fault isolation is
       the real safety net.  Land P2 only after Phase D is in.

    3. P4 (threading) tends to introduce new races during the
       transition.  Mitigation: validate each migrated subsystem with
       ThreadSanitizer (Linux build for testing only, even though
       production is Windows).  Or: skip P4 if P1 already delivers
       enough stability.

  Fallback if 6 weeks is too long:

    The 80/20 of this plan is P0 + P1 + P5 + P6.  Those four items
    address >80% of the crash root causes and can be done in 2-3
    weeks.  P2/P3/P4 are about long-term maintainability rather than
    raw stability — defer them if user-visible stability is the goal
    and the current crash rate post-P1 is acceptable.

  Fallback if even 3 weeks is too long:

    P5 (concurrency bounds) + P6 (validation layers) alone, in two
    days.  Reduces TDR rate substantially, surfaces remaining bugs
    cleanly, costs almost nothing.  Not a long-term fix but a useful
    bridge.

ONE-PARAGRAPH CLOSING
────────────────────────────────────────────────────────────────────────────────

  This is a fundamentally salvageable project.  The shader code is good,
  the Spine integration works, the UI is functional, the export pipeline
  ships.  The crashes come from the seam between subsystems, not from
  inside any single subsystem.  Centralize GPU submission, finish the
  render graph that's already 60% done, collapse the caches, bound the
  worker counts.  Don't rewrite — consolidate.  The biggest favor you
  can do this codebase is delete 2500 lines.
