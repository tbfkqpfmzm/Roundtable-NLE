3. REMAINING ARCHITECTURAL DEFECTS
================================================================================

  3.1 Heavy Sub-TUs Around CompositeService
  ────────────────────────────────────────────
  Severity: HIGH
  Status: 🟢 CompositeEngine extracted — GPU pipeline logic moved out.
  Remaining heavy TUs that still hold significant inline logic:
    - CompositeServiceLayerBuild.cpp (1271 lines) — timeline traversal
      for layer building
    - CompositeServiceFrame.cpp (~1036 lines) — compositeFrame
      orchestration, prewarm, spine rendering

  Impact: These sub-TUs are now the main untested surface. Layer-building
  or frame-orchestration bugs can still crash the compositing pipeline.

  3.2 Thread Model Is Still Broken
  ──────────────────────────────────
  Severity: CRITICAL
  Status: ❌ Not addressed
  compositeFrame() still uses the try_to_lock pattern:
    - On contention: returns stale m_lastGoodComposite
    - 6+ early-exit paths return stale frames instead of blocking
    - Audio clock still tightly coupled to GPU compositing

  Impact: Frame-accurate seeking and A/V sync drift correction are
  fundamentally unreliable until this is fixed.

  3.3 GPU Pipeline Still Single-Threaded
  ─────────────────────────────────────────
  Severity: HIGH
  Status: 🟡 Triple-buffering fixed command buffer corruption, but
  the pipeline still records ALL work (uploads, effects, transitions,
  composite, readback) into ONE command buffer per frame. No parallelism.
  No render graph. No fault isolation between passes.

  3.4 Three Incoherent Caches
  ─────────────────────────────
  Severity: MEDIUM
  Status: ❌ Not addressed
  CPU FrameCache, GpuTextureCache, and composite LRU cache all have
  independent eviction policies. No unified budget. No VRAM pressure
  monitoring. Loop frames and pinned frames accumulate unbounded.

6. RECOMMENDED LONG-TERM VISION (2-3 Months)
================================================================================

  The render graph rewrite (Phase 6 of RENDER_FIX_PLAN.txt) is the correct
  long-term architecture. It should only be attempted after Phases 1-5 are
  complete and stable.

  Recommended timeline:
    - Month 1: Complete all short-term + medium-term priorities above
    - Month 2: Design + implement render graph DAG, triple-buffered frame
      pipeline, and unified resource management
    - Month 3: Replace try_to_lock with proper MPMC frame queue, add
      telemetry, remove CPU fallback path dependency

  The render graph architecture:
    ┌──────────────────────────────────────────────────────────────┐
    │  RENDER GRAPH (DAG of render passes)                        │
    │  Each pass: input resources → shader → output resources     │
    │  Pass types: Decode, Upload, Effect, Transition, Composite  │
    │  Automatic pipeline barriers between passes                 │
    │  Fault isolation: one failed pass doesn't crash the frame   │
    │                                                            │
    │  FRAME PIPELINE (triple-buffered)                           │
    │  Frame N:   recording (CPU)                                 │
    │  Frame N-1: executing (GPU)                                 │
    │  Frame N-2: fence signaled, resources available             │
    │                                                            │
    │  PROPER THREAD MODEL                                        │
    │  FrameClock → Lock-free MPMC queue → Composite Workers      │
    │  NEVER return "stale frame" — always produce correct frame  │
    │  Backpressure: skip clock tick when queue full              │
    │                                                            │
    │  UNIFIED RESOURCE MANAGEMENT                                │
    │  Single cache with generation tracking                      │
    │  Fence-aware eviction (only after GPU signals)              │
    │  VRAM pressure monitoring with emergency eviction           │
    └──────────────────────────────────────────────────────────────┘
