PARSER_PLAN v3 — .shotanim Native Runtime (Seamless In-App Conversion)
================================================================================

*** STATUS: DEFERRED (2026-05-15) ***

This effort is on hold. The original motivation was to avoid shipping
spine-cpp. That motivation no longer holds: the project owner has a Spine
Editor license, and Section 2 of the Spine Editor License Agreement
permits shipping the Spine Runtimes inside ROUNDTABLE (a video NLE that
adds significant primary functionality on top of the runtimes). spine-cpp
is therefore the production renderer and ROUNDTABLE can ship today.

The native-runtime work (writer + reader + evaluator + parity/migrate
tools) has been moved out of the build to:

    tools/native_engine/

See tools/native_engine/README.md for the current state, the remaining
blocker (path constraints are entirely unimplemented — every Nikke rig
uses them), and step-by-step instructions to re-enable and resume.

License notice: docs/THIRD_PARTY_LICENSES.md and the repo-root NOTICES.md
now reproduce the Spine Runtimes License Agreement verbatim, satisfying
Section 2.2(a) of the Spine Editor License Agreement.

The remainder of this document is the original plan, preserved for
historical context and for whoever resumes the work.

Last updated: 2026-05-15

Goal
----
Replace spine-cpp's RUNTIME role with a native .shotanim runtime that plays
back our own animation format. Spine-cpp remains in the editor strictly as
a one-shot IMPORT library — invoked the instant a character is downloaded
from Nikke-db, then never touched again for that character.

User experience (the critical UX promise)
-----------------------------------------
End users never see "Spine" or ".skel" anywhere. They open the Character
Browser, click Download, and get a playable character. Internally:

    Download .skel + .atlas + .png from Nikke-db
        ↓
    Convert .skel → .shotanim using spine-cpp (one shot, invisible)
        ↓
    Delete the original .skel
        ↓
    Persist character.shotanim + skeleton.png on disk
        ↓
    All future loads: .shotanim → NativeSpineEngine → render
    (spine-cpp never invoked again for that character)

This means:
  - The editor binary keeps shipping spine-cpp during the transition.
  - End users do nothing different from today.
  - The runtime hot path is 100% our code (NativeSpineEngine reads .shotanim).
  - Eventually, we can replace spine-cpp's import role with our own .skel
    parser or with SpineSkeletonDataConverter — that decision can wait
    until the runtime is proven at parity.

Dual-path during validation
---------------------------
Two CMake gates run side-by-side:

    ROUNDTABLE_HAS_SPINE              (always ON during transition)
        Compiles spine-cpp; used for both runtime (legacy) and the
        download-time conversion step.

    ROUNDTABLE_HAS_NATIVE_SHOTANIM    (OFF by default; ON once parity verified)
        Compiles ShotAnimReader + NativeSpineEngine.
        When ON, the editor PREFERS .shotanim if present, else falls
        back to the spine-cpp runtime path.

Migration arc:
  Today:   spine-cpp runtime, no .shotanim, no native engine
  Phase A: Add writer + reader + native engine, gated OFF, validate parity
  Phase B: Wire auto-conversion into download flow, flip flag ON
  Phase C: Spine-cpp keeps shipping but only used at download time
  Phase D (future): Replace import-time spine-cpp with our own .skel parser
                    or SpineSkeletonDataConverter — only when convenient


================================================================================
KEY INSIGHT — We Don't Write a .skel Parser
================================================================================

The open-source project `SpineSkeletonDataConverter` provides:

  - Custom, from-scratch .skel binary readers for Spine 3.5–4.2
  - Custom, from-scratch .skel binary writers for Spine 3.5–4.2
  - A version-agnostic SkeletonData intermediate struct containing all
    bone, slot, skin, attachment, and animation data
  - No dependency on spine-cpp for skeleton data parsing
  - License: PolyForm Noncommercial 1.0.0

This means: we do NOT write a .skel binary parser. Instead, we write a
SkeletonData → .shotanim converter that plugs into this project's output.

    SpineSkeletonDataConverter        Our code            Editor (distributed)
    ───────────────────────────        ────────            ────────────────────
    .skel ──→ SkeletonData ──→ .shotanim writer ──→ .shotanim blob
                                                        │
                                                        ▼
                                               .shotanim reader + runtime
                                               (our code, ~1100 lines)

The converter is a developer tool. It is NEVER linked into the editor binary.
Editor users either download pre-converted .shotanim files (SPINE_FALLBACK_PLAN)
or run the converter locally on their own .skel files.

SECTION 2 — Legal Posture (Updated)
================================================================================

What the distributed editor contains:
  - .shotanim reader: reads OUR binary format. We defined every byte.
  - Native evaluator: samples keyframes, interpolates, resolves hierarchy,
    emits render commands. Standard computer graphics math — no novel
    algorithms, no copied code.
  - NO .skel parsing code. NO spine-cpp code. NO Esoteric code of any kind.

What the developer converter tool contains:
  - SpineSkeletonDataConverter (open source, PolyForm Noncommercial).
    Used ONLY as a developer build tool. Not distributed to end users.
    Not linked into the editor binary.
  - .shotanim writer: OUR code. Reads the SkeletonData struct (generic
    data: bone names, transforms, keyframe arrays) and writes our format.
    No algorithmic derivation from spine-cpp.

Why this is legally sound:
  1. The editor binary contains ZERO third-party Spine code. We own 100%
     of the code that ships.
  2. The .shotanim format is our own design — we defined the header,
     field layout, encoding, and semantics.
  3. The runtime evaluator performs standard math operations on our own
     data structures. Keyframe interpolation, matrix multiplication,
     and hierarchy resolution are foundational computer graphics — not
     something Esoteric can claim ownership of.
  4. The developer converter uses an open-source tool that implements
     its own .skel readers (not derived from spine-cpp). The converter
     is an internal build tool, analogous to using ImageMagick to
     pre-process textures before packaging.

Risk comparison to current state:
  Current: spine-cpp is distributed in every user binary.
           → Every user needs a Spine Editor license (per spine-cpp license).
           → Clear contractual violation.
  New:     Zero Esoteric code in distributed binary.
           → No contractual relationship with Esoteric.
           → Our format, our runtime, our code.

SECTION 3 — The .shotanim Format
================================================================================

A single binary blob containing everything needed for playback of one
character/outfit/stance combination (all animations, all skins).

File layout:

  ┌────────────────────────────────────────────┐
  │  HEADER                                    │
  │    magic:        "SHTA" (4 bytes)          │
  │    version:      uint32 (1)                │
  │    boneCount:    uint16                     │
  │    slotCount:    uint16                     │
  │    skinCount:    uint16                     │
  │    animCount:    uint16                     │
  │    pageCount:    uint16                     │
  │    skeletonWidth:  float                    │
  │    skeletonHeight: float                    │
  │    flags:        uint32                     │
  │      bit 0: hasClipping                    │
  │      bit 1: hasMeshAttachments             │
  │      bit 2: hasDrawOrderKeyframes          │
  │    reserved:     28 bytes                   │
  │  (total: 64 bytes)                         │
  ├────────────────────────────────────────────┤
  │  STRING TABLE                              │
  │    count:       uint32                     │
  │    [count] entries:                        │
  │      length:    uint16                     │
  │      data:      char[length]               │
  │  (bone names, slot names, attachment       │
  │   names, animation names, skin names,      │
  │   texture paths — deduplicated)            │
  ├────────────────────────────────────────────┤
  │  BONE HIERARCHY                            │
  │    [boneCount] entries:                    │
  │      nameIdx:     uint16 (string table)    │
  │      parentIdx:   int16  (-1 = root)       │
  │      restX:       float                    │
  │      restY:       float                    │
  │      restRotation: float                   │
  │      restScaleX:  float                    │
  │      restScaleY:  float                    │
  │      restShearX:  float                    │
  │      restShearY:  float                    │
  │      length:      float                    │
  │  (38 bytes per bone, in parent→child       │
  │   topological order)                       │
  ├────────────────────────────────────────────┤
  │  TEXTURE PAGES                             │
  │    [pageCount] entries:                    │
  │      pathIdx:    uint16 (string table)     │
  │      width:      uint16                    │
  │      height:     uint16                    │
  │      pma:        uint8  (0/1)              │
  │      padding:    1 byte                    │
  │  (8 bytes per page)                        │
  ├────────────────────────────────────────────┤
  │  DEFAULT SKIN SLOTS                        │
  │    [slotCount] entries:                    │
  │      slotNameIdx:    uint16                │
  │      boneIdx:        uint16                │
  │      blendMode:      uint8                 │
  │      attachmentNameIdx: uint16 (or 0xFFFF) │
  │  (7 bytes per slot)                        │
  ├────────────────────────────────────────────┤
  │  REGION ATTACHMENTS (pool)                 │
  │    count:       uint32                     │
  │    [count] entries:                        │
  │      nameIdx:      uint16 (string table)   │
  │      pageIdx:      uint16                  │
  │      x:            float                   │
  │      y:            float                   │
  │      width:        float                   │
  │      height:       float                   │
  │      u:            float                   │
  │      v:            float                   │
  │      u2:           float                   │
  │      v2:           float                   │
  │      offsetX:      float                   │
  │      offsetY:      float                   │
  │      originX:      float                   │
  │      originY:      float                   │
  │      rotation:     float                   │
  │      colorR:       float                   │
  │      colorG:       float                   │
  │      colorB:       float                   │
  │      colorA:       float                   │
  │  (76 bytes per region)                     │
  ├────────────────────────────────────────────┤
  │  MESH ATTACHMENTS (pool)                   │
  │    count:       uint32                     │
  │    [count] entries:                        │
  │      nameIdx:      uint16                  │
  │      pageIdx:      uint16                  │
  │      vertexCount:  uint16                  │
  │      triangleCount:uint16                  │
  │      hullLength:   uint16                  │
  │      padding:      2 bytes                 │
  │      vertices:   float[vertexCount * 2]    │
  │      uvs:        float[vertexCount * 2]    │
  │      triangles:  uint16[triangleCount]     │
  │      edges:      uint16[vertexCount]       │
  │      hull:       uint16[hullLength]        │
  │      colorR/G/B/A: float (4)               │
  │      originX/Y:    float (2)               │
  │  (variable size per mesh)                  │
  ├────────────────────────────────────────────┤
  │  SKINS (beyond default)                    │
  │    [skinCount-1] entries:                  │
  │      skinNameIdx:  uint16                  │
  │      slotCount:    uint16                  │
  │      [slotCount] entries:                  │
  │        slotNameIdx:     uint16             │
  │        attachmentNameIdx: uint16           │
  │  (variable size per skin)                  │
  ├────────────────────────────────────────────┤
  │  ANIMATIONS                                │
  │    [animCount] entries:                    │
  │                                                  │
  │      ANIM HEADER                                │
  │        nameIdx:      uint16                     │
  │        duration:     float                      │
  │        timelineCount:uint16                     │
  │        padding:      2 bytes                    │
  │      (12 bytes)                                 │
  │                                                  │
  │      BONE TIMELINES [one per animating bone]     │
  │        boneIdx:      uint16                      │
  │        property:     uint8                       │
  │          0=rotate, 1=translateX, 2=translateY    │
  │          3=scaleX, 4=scaleY, 5=shearX, 6=shearY  │
  │        interpolation: uint8                      │
  │          0=linear, 1=stepped, 2=bezier            │
  │        keyframeCount: uint16                     │
  │        [keyframeCount] entries:                  │
  │          time:    float                          │
  │          value:   float                          │
  │          (if bezier: inTangent, outTangent)      │
  │                                                  │
  │      SLOT TIMELINES                              │
  │        slotIdx:    uint16                        │
  │        property:   uint8                         │
  │          0=attachment, 1=colorR, 2=colorG        │
  │          3=colorB, 4=colorA                       │
  │        interpolation: uint8                      │
  │        keyframeCount: uint16                     │
  │        [keyframeCount] entries:                  │
  │          time:   float                           │
  │          value:  float (or string table idx)     │
  │                                                  │
  │      DRAW ORDER TIMELINE (if flags bit 2)        │
  │        keyframeCount: uint16                     │
  │        [keyframeCount] entries:                  │
  │          time:   float                           │
  │          offsets: uint16[slotCount]              │
  │                                                  │
  │      ANIM FOOTER (end marker = 0xFFFF)           │
  ├────────────────────────────────────────────┤
  │  FOOTER                                    │
  │    crc32:   uint32 (of all above data)     │
  │    magic:   "ENDA" (4 bytes)               │
  └────────────────────────────────────────────┘

SECTION 4 — Developer Converter: SkeletonData → .shotanim
================================================================================

4.1  Tool location in codebase

  tools/shotanim_writer.cpp          (new, ~400 lines)
  tools/shotanim_writer.h            (new)
  tools/convert_character.ps1        (new, batch script)

  External dependency:
    wang606/SpineSkeletonDataConverter  (git submodule, developer builds only)

4.2  Input

  A character directory containing:
    skeleton.skel
    skeleton.atlas
    skeleton.png
    (aim/skeleton.skel, aim/skeleton.atlas, aim/skeleton.png)
    (cover/skeleton.skel, cover/skeleton.atlas, cover/skeleton.png)

4.3  Conversion pipeline

  For each variant (default, aim, cover):

    Step 1: .skel → SkeletonData
      SpineSkeletonDataConverter reads the .skel binary into its
      version-agnostic SkeletonData struct.
      Command: SpineSkeletonDataConverter.exe skeleton.skel temp.json
      (We can also use the library API directly to avoid JSON round-trip.)

    Step 2: .atlas → AtlasData
      Parse the .atlas text file into page dimensions, region UV coordinates,
      and texture paths. Simple line-by-line parser. (~150 lines)

    Step 3: SkeletonData + AtlasData → .shotanim
      Walk the SkeletonData struct:
        bones[]          → bone hierarchy table
        slots[]          → slot definition table
        skins[]          → skin/attachment tables
        animations[]     → keyframe arrays
        AtlasData.pages  → texture page table
        AtlasData.regions → UV coordinates for attachments

      Write the binary blob per the .shotanim format spec (Section 3).

    Step 4: Copy .png texture files as-is.

4.4  Key SkeletonData fields we use

  From SpineSkeletonDataConverter's include/SkeletonData.h:

    struct SkeletonData {
        std::string version;
        float width, height;
        std::vector<BoneData> bones;
        std::vector<SlotData> slots;
        std::vector<SkinData> skins;
        std::vector<AnimationData> animations;
    };

    struct BoneData {
        OptStr name;
        OptStr parent;        // name of parent (empty for root)
        float x, y;           // rest position
        float rotation;       // rest rotation (degrees)
        float scaleX, scaleY; // rest scale
        float length;         // bone length
    };

    struct SlotData {
        OptStr name;
        OptStr bone;           // parent bone name
        OptStr attachment;     // default attachment name
        Color color;           // RGBA tint
        OptStr blendMode;      // "normal", "additive", "multiply", "screen"
    };

    struct AnimationData {
        OptStr name;
        float duration;
        std::vector<TimelineData> timelines;
    };

    struct TimelineData {
        int boneIndex;         // or slotIndex
        OptStr property;       // "rotate", "translateX", etc.
        std::vector<TimelineFrame> frames;
    };

    struct TimelineFrame {
        float time;
        float value;
        OptStr stringValue;    // for attachment keyframes
        CurveType curve;       // linear, stepped, bezier
        float inTangent, outTangent;
    };

  All of this maps directly to .shotanim sections. No creative
  transformation needed — just data reorganization.

4.5  Batch conversion script

  convert_character.ps1:
    param($inputDir, $outputDir)
    For each variant (default, aim, cover):
      if .skel exists:
        .\SpineSkeletonDataConverter.exe $skelPath temp.json
        .\shotanim_writer.exe temp.json $atlasPath $outputPath.shotanim
        Copy .png to $outputDir
        Remove temp.json

  convert_all_characters.ps1:
    Walk assets/characters/ directory
    For each character/outfit/stance:
      convert_character.ps1 -inputDir $dir -outputDir $dir

SECTION 5 — Editor-Side: .shotanim Reader + Runtime
================================================================================

5.1  New source files (distributed binary)

  src/core/spine/ShotAnimReader.h       (~80 lines)
  src/core/spine/ShotAnimReader.cpp     (~300 lines)
  src/core/spine/NativeSpineEngine.h    (~100 lines)
  src/core/spine/NativeSpineEngine.cpp  (~800 lines)

5.2  ShotAnimReader — binary deserializer

  Loads a .shotanim file into memory-resident structs:

    struct ShotAnimData {
        Header header;
        std::vector<std::string> strings;
        std::vector<BoneDef> bones;
        std::vector<TexturePage> pages;
        std::vector<SlotDef> slots;
        std::vector<RegionAttachment> regions;
        std::vector<MeshAttachment> meshes;
        std::vector<Skin> skins;
        std::vector<Animation> animations;
    };

  Simple sequential read — no alloc in hot path (done once at load time).
  Validates header magic + CRC32 footer. Returns error on corruption.

5.3  NativeSpineEngine — per-frame evaluator

  Mirrors current SpineEngine API exactly:

    bool loadSkeleton(const std::string& shotanimPath);
    bool loadFromClip(const SpineClip& clip, const std::string& assetsDir);
    bool setSkin(const std::string& skinName);
    std::vector<std::string> listSkins() const;
    std::vector<AnimationInfo> listAnimations() const;
    void evaluateAtTime(float bodyTime, float talkTime = 0.0f);
    SpineRenderData extractMeshes();
    void getBounds(float& x, float& y, float& w, float& h);

  Per-frame evaluation algorithm:

    1. Find animation by name (string table lookup)
    2. Clamp/wrap time to animation duration
    3. For each bone timeline:
       a. Binary search keyframes → prev/next frame
       b. Interpolate value (lerp for linear, no-interp for stepped)
       c. Build local transform matrix (translate × rotate × scale × shear)
    4. Resolve world transforms (walk bones in parent→child order,
       guaranteed by .shotanim layout — parent always at lower index)
    5. For each slot timeline:
       a. Sample attachment keyframes → current attachment name
       b. Sample color keyframes → interpolated slot color
    6. Resolve draw order (sample draw order timeline or use default)
    7. For each active slot in draw order:
       a. Look up attachment in current skin → default skin
       b. Skip if no attachment or alpha < epsilon
       c. If clipping: handle clip start/end (push/pop clipper state)
       d. If region: compute 4 world vertices from bone + region offsets
       e. If mesh: compute N world vertices from bone + mesh vertex data
       f. Combine skeleton color × slot color × attachment color
       g. Batch by (pageIndex, blendMode), merging only with last batch
          to preserve draw order
    8. Compute bounding box from world vertices

  Memory discipline:
    - World transform array: pre-allocated at load time (boneCount × 16 floats)
    - Temp vertex/UV/clip arrays: pre-allocated at load time
    - Zero heap allocations in the evaluation loop
    - Binary search: O(log n) per keyframe lookup
    - No std::map, no std::unordered_map, no string operations in hot path

5.4  Scrubbing support

  evaluateAtTime() takes an absolute time. No state carries between frames.
  Scrubbing is instantaneous — just call with the new time.

5.5  Talking / multi-track support

  Simplified from current spine-cpp approach:
    - Body animation: always on Track 0
    - Talk animation: on Track 1 when m_talking is true
    - Talk blending: evaluate both tracks, apply talk values additively
      to the pose (talk bone transforms are relative to rest pose)
    - Talk auto-detection: same priority list as current code
      (talk_start > contains "mouth" > talk_loop > any with "talk")

SECTION 9 — Testing Strategy
================================================================================

9.1  Round-trip fidelity
     For each character/outfit/stance:
       1. Old path: spine-cpp → SpineRenderData at every 10th frame
       2. New path: .shotanim → SpineRenderData at every 10th frame
       3. Compare: vertex positions, UVs, indices, colors — all within
          floating-point epsilon (1e-4 for positions, exact for indices)

9.2  Performance
     - .skel → .shotanim conversion: target < 500ms per character
     - .shotanim → memory load: target < 50ms
     - Per-frame evaluation: target ≤ current spine-cpp path
     - Memory: .shotanim footprint (50KB–2MB) vs. spine-cpp footprint
       (similar, since it's the same structural data)

9.3  Edge cases
     - Empty skeleton (0 bones)
     - Single-bone skeleton, single-keyframe animation
     - Very long animations (60s+)
     - Animation with 0 keyframes → no-op, not crash
     - Missing texture page → graceful error
     - Corrupted .shotanim (bad CRC32) → graceful error, not crash
     - .shotanim version mismatch → clear error message
     - Multiple texture pages per character → all render correctly
     - Scrubbing backwards across clip boundaries

9.4  Regression
     - All existing tests in tests/test_spine.cpp must pass
     - All existing tests in tests/test_gpu_spine.cpp must pass
     - All existing tests in tests/test_clips.cpp must pass
     - Export output must be pixel-identical to current export path

SECTION 10 — Relationship to SPINE_FALLBACK_PLAN
================================================================================

This plan and SPINE_FALLBACK_PLAN share the exact same runtime — the
.shotanim reader + native evaluator is identical in both. They differ
only in how .shotanim files are produced and distributed:

    PARSER_PLAN v2                     SPINE_FALLBACK_PLAN
    ────────────────                   ───────────────────
    Converter runs on user's           Converter runs on developer's
      machine                            machine, uploaded to GitHub
    User downloads .skel from          User downloads .shotanim from
      Nikke-db GitHub                   OUR GitHub
    .shotanim generated locally        .shotanim pre-generated + hosted
    User controls conversion           Developer controls conversion
    No character hosting by us         We host character derivatives
    Users always get latest from       Slight delay for new characters
      Nikke-db directly                 (must run converter first)

The plans are compatible and composable. You can:
  - Build the shared runtime first
  - Support BOTH modes: download .shotanim if available on our host,
    otherwise download .skel from Nikke-db and convert locally
  - Decide the final distribution model later

SECTION 11 — Open Questions / Risks
================================================================================

1. SpineSkeletonDataConverter maintenance
   Supports Spine 3.5–4.2. If Nikke-db characters move to a newer
   Spine version, the converter must be updated. As an open-source
   project under noncommercial license, updates may lag.
   Mitigation: The architecture (version-specific readers feeding a
   common SkeletonData struct) makes adding new versions straightforward.
   We could contribute version readers upstream if needed.

2. Atlas parsing
   SpineSkeletonDataConverter includes spine-cpp's Atlas.cpp for .atlas
   parsing. For our converter, this is acceptable (converter is not
   distributed). Alternatively, we write a simple text .atlas parser
   (~150 lines) to eliminate the dependency entirely.

3. Attachment type coverage
   Must handle: region, mesh, linkedmesh, clipping.
   Can skip: boundingbox, path, point (no visual output).
   LinkedMesh requires resolution to parent mesh — handle in writer.

4. Curve type normalization
   Spine 3.x and 4.x encode bezier curves differently (absolute vs.
   relative control points). SpineSkeletonDataConverter normalizes this
   into a common SkeletonData representation. We use the normalized form
   in .shotanim.

5. PolyForm Noncommercial license
   The converter tool cannot be distributed commercially. Since it's
   used only as a developer build tool (never shipped to end users),
   this is fine. Our .shotanim runtime has no license restrictions — we
   own it entirely.

6. IK constraints
   If Nikke characters use Inverse Kinematics, we must implement an IK
   solver in the runtime evaluator. This is the most complex feature.
   The Phase 1 audit (via compare_parsers) should determine if IK is
   present in our assets. If not, this can be deferred indefinitely.
